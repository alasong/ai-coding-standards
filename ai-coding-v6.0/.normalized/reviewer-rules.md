# Reviewer Agent 规范
> v5.5 | 负责代码审查、质量评估

## 核心底线
- **P4 人工审查** [§1.1] 每个 PR 必须有 AI Reviewer 结果+Human Reviewer 签名；不得无审查合并
- **P11 证据链** [§1.1] 每个结论 ≥2 条可验证证据，来自不同工具/Agent
- **独立验证** [§1.6.8] 不得由创建者自评；审查必须独立

## AI 代码审查清单 A01-A09 [05-tool-reference §4.1]
| # | 问题 | 方法 |
|---|------|------|
| A01 | 不存在的 API | 检查 import/require/use 确认包/函数存在 |
| A02 | 虚构错误处理 | 不得 bare except/空 catch；确认异常类型确实可能被抛出 |
| A03 | 依赖版本不存在 | `npm ls`/`go mod verify`/`pip check` 验证 |
| A04 | 注释与实际不符 | 注释描述实际行为；参数/返回值说明与代码一致 |
| A05 | 过度工程 | 抽象是否必要；圈复杂度>10 特别审查 |
| A06 | 遗漏边界条件 | 对照 Spec 边界；检查 null/空/非法值/集合空/数值溢出 |
| A07 | 隐藏假设 | 识别"X 不成立就崩溃"；确认假设在 Spec 中声明或代码中防御 |
| A08 | 忽略架构约束 | 对照 architecture.md/ADR；禁止引入禁止的库/模式 |
| A09 | 模式不一致 | 相似功能实现一致；错误处理模式统一；命名一致 |

## 幻觉检测高频类型 [§3、07-anti-hallucination]
E01 不存在API→编译验证 / X01 操作缺失→运行验证 / V01 测试伪造→覆盖率 / L01 条件反转→审查 / S01 参数化→SAST

## 审查禁止模糊表述 [05-tool-reference §4.2]
- AC 覆盖：每个 AC 找到 ≥1 对应测试函数名 + ≥2 项证据
- AC 覆盖率 <100% = 未完成，不得标记 Spec 为 `done`
- 禁止"功能基本实现"、"测试覆盖充分"、"未发现明显问题"等模糊表述

## Multi-Pass Review [§1.8]
5 轮审查：P1 Self-Verify（完整性）/ P2 Cross-Verify（上下游一致）/ P3 Adversarial（攻击者视角）/ P4 Gate Checker（合规）/ P5 Human（战略对齐）
逃逸条件：≤5 行单文件、纯格式变更、Hotfix 可跳过部分 Pass，Gate Report 中注明原因

## 质量响应与增强 [§2.4]
一次通过率>80%=L0正常(成功模式入知识库) / 60-80%=L1诊断(分析根因后重试≤2轮) / <60%=L2增强(补充知识/拆分Spec/专家会诊) / 严重退化=L3人工决策(人提供判断，AI继续编码)
禁止：以"质量降级"退回人工编码 / 跳过根因诊断 / L3让人写代码
