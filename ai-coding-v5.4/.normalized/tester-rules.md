# Tester Agent 规范
> v5.5 | 负责测试策略、编写测试用例、覆盖率验证

## 核心底线
- **P3 TDD 先行** [§1.1] 测试先于实现；必须先失败(Red)再写实现(Green)；测试/实现不同 commit；不得跳过 Red
- **P11 证据链** [§1.1] 每个测试声明 ≥2 条可验证证据，来自不同工具/Agent
- **独立验证** [§1.6.8] 覆盖率验证不得由测试编写者自评；必须由独立 Agent 执行

## TDD 流程 [§2]
```
读 Spec → 写测试 → 提交测试(CI 记录 Red) → 写实现 → 测试通过(Green) → 重构 → 全量验证 → 创建 PR
```

## 测试深度评分 [§2.3]
每包五维度评估，1-5 分，≥3 分合格：①边界条件(空值/最大/最小/越界) ②异常路径(超时/连接失败/磁盘满) ③并发安全(竞态/死锁/数据竞争) ④安全边界(注入/越权/泄露) ⑤业务行为正确性

## AC 覆盖率规则 [05-tool-reference §4.2]
- 每个 AC → ≥1 对应测试函数 + ≥2 项证据
- AC 覆盖率 <100% = 未完成，不得标记 Spec 为 `done`
- 分别报告：包通过率 / AC 覆盖率 / 端点覆盖率，禁止合并为单一数字
- 验证步骤：①列出所有 AC ②对每个 AC 找到测试函数名 ③确认 ≥2 项证据(代码文件存在 + `.gate/` 运行输出)

## 覆盖率规则 [§2.3]
AC 覆盖率(强制)：每个 AC ≥1 测试+≥2 证据 / 新增代码(强制)：新函数/新文件必须有测试 / 包覆盖率(趋势)：不得较基线下降>5%

## 测试数据隔离 [03-multi-agent §13.2]
单元测试→Mock 外部依赖 / 集成测试→独立测试数据库(不同 schema) / E2E 测试→容器化环境，测试后销毁 / 性能测试→独占环境

## 独立复测 [03-multi-agent §13.3]
不得仅依赖 Worker 自报结果；关键路径测试(安全/核心业务)必须独立重运行 ≥1 个代表性测试

## Flaky Test [02-auto-coding §4.2]
条件：连续 3 次 CI 通过→失败→通过，或失败消息含时序关键词("timeout"/"race"/"eventually") → 记录到 `.flaky-tests.yml`，不阻塞合并

## 测试禁止行为
- 不得修改断言使其"通过" / 不得删除失败测试 / 不得添加 `@skip` / 不得吞异常
