## 第 1 章：核心原则

### 1.1 核心原则清单（P1-P11，不可违反）

| # | 原则 | 说明 | Auto-Coding 含义 |
|---|------|------|-----------------|
| **P1** | **商业驱动** | 所有开发活动必须有明确的商业目标支撑 | 无商业目标的 AI 编码是纯浪费 |
| **P2** | **DCP 门禁** | 每个阶段必须通过决策门才能进入下一阶段 | L3/L4 可异步执行，但不得跳过 |
| **P3** | **TDD 先行** | AI 生成实现前必须先有测试，且测试必须先失败（Red） | L3/L4 下 AI 自动执行 Red→Green→Refactor，CI 必须记录 Red 状态 |
| **P4** | **人工审查** | 所有 AI 生成的代码必须经过人工 Code Review 才能合并 | L4 仅 trivial fix 可自动合并 |
| **P5** | **密钥不入代码** | 任何密钥、密码、token 不得出现在代码或配置文件中 | 必须配置 pre-commit hook + SAST 双重拦截 |
| **P6** | **单一信息源** | 每个事实只在一个地方定义，其他地方引用 | 过时文档=AI 按旧信息编码 |
| **P7** | **Spec 驱动** | AI 生成代码必须有明确的 Spec 文件作为输入 | L3/L4 下 Agent 直接从 `specs/` 读取任务队列 |
| **P8** | **最小批量** | AI 一次只生成一个函数或小模块的代码 | >50 行函数或 >200 行文件必须拆分 |
| **P9** | **Prompt 版本化** | 用于生成代码的 Prompt 必须版本化并可追溯 | L3/L4 下自动持久化到 `prompts/` |
| **P10** | **数据分级** | 发送给 AI 的数据必须经过分类，敏感数据禁止发送 | 必须实现 pre-send 扫描自动拦截 |
| **P11** | **证据链** | 每个 AI 声明必须有可验证的证据支撑，机器可验证、跨引用≥2 条、证据链不断裂 | L2+ 自动收集到 `.gate/` 目录 |

### 1.2 工程实践原则（P12-P23）

| # | 原则 | 说明 |
|---|------|------|
| **P12** | **环境一致性** | 开发环境必须通过声明式配置管理，不靠运行时重试 |
| **P13** | **错误不吞** | 所有错误必须处理或返回，禁止 `//nolint:errcheck` |
| **P14** | **租户隔离** | 禁止硬编码租户 ID，从请求上下文提取 |
| **P15** | **并发安全** | 全局可变状态必须使用锁或原子操作保护 |
| **P16** | **资源清理** | 所有 map/channel/goroutine 必须有清理机制 |
| **P17** | **输入校验** | 所有外部输入视为不可信，必须验证 |
| **P18** | **JSON 安全** | SSE/HTTP 响应数据必须用 `json.Marshal`，禁止拼接 |
| **P19** | **认证门禁** | 所有写操作端点必须有认证和角色校验 |
| **P20** | **速率保护** | 所有昂贵端点必须有速率限制 |
| **P21** | **数据一致性** | DB 写入失败必须返回错误，不得静默降级 |
| **P22** | **IP 不暴露** | 生产 IP/域名不得出现在代码中 |
| **P23** | **需求→Spec 链** | AI 生成代码前必须经过需求分析→架构适配→方案设计→Spec 生成四阶段链 |

### 1.2.1 工程实践原则详解

| 原则 | 违反后果 | Auto-Coding 含义 |
|------|---------|-----------------|
| P12 环境一致性 | 开发者无法启动、CI 失败 | 环境配置必须声明式、可复现、语言适配 |
| P13 错误不吞 | 运行时静默失败 | 禁止 `//nolint:errcheck`、`_ = json.Marshal(...)` |
| P14 租户隔离 | 数据泄漏、多租户污染 | 代码中不得出现硬编码租户字符串 |
| P15 并发安全 | 数据竞争、不可复现 Bug | 任何被多个 goroutine 读写的变量必须加锁 |
| P16 资源清理 | 内存泄漏、goroutine 泄漏 | 全局缓存必须设 TTL/上限并启动后台清理 |
| P17 输入校验 | 400 请求被接受导致异常 | 不得信任调用方输入 |
| P18 JSON 安全 | 无效 JSON 导致客户端解析失败 | SSE `data:` 后面必须是合法 JSON |
| P19 认证门禁 | 未授权数据修改 | GET 可跳过认证，POST/PUT/DELETE 必须有角色校验 |
| P20 速率保护 | DoS 攻击、资源耗尽 | 无 rate limiter 的端点不得上线 |
| P21 数据一致性 | 数据不一致，调用方误以为成功 | 不得用空返回值掩盖写失败 |
| P22 IP 不暴露 | 配置泄漏、环境切换困难 | 生产地址必须走 `.env` 或 `config.yaml` |

#### 多层执行机制（Layered Enforcement）

> **语言无关性声明**：P12-P22 是语言无关的原则。表格中的 Go 代码示例（如 `//nolint:errcheck`、`goroutine`）仅为 Go 语言下的具体违规表现。在 Python/TypeScript 等其他语言中，同等原则的违规表现为该语言的等效模式（如 Python 中的 `except: pass`、TypeScript 中的 `.catch(() => {})` 空处理）。AI 应将每条原则翻译为当前项目语言的等效检查和防护。

P12-P22 不是靠 AI 自觉执行，而是通过四层机制自动拦截：

| 层级 | 机制 | 示例 | 拦截能力 |
|------|------|------|---------|
| **L1 AI 约束** | 规范声明 | AGENTS.md 中 P13-P22 声明 | 弱——AI 可能遗忘 |
| **L2 Pre-commit** | Git hook 自动拦截 | 拦截 `//nolint:errcheck` | 强——提交时阻断 |
| **L3 CI Gate** | CI 门禁 | `go test -race`、lint | 强——CI 阻断 |
| **L4 Runtime** | 代码本身防护 | `writeJSON` 返回 500 | 强——运行时兜底 |

**执行原则**：一条规则至少有两层保护（L1 + L2/L3/L4 之一）。只有 L1 的规则视为"未落地"。

### 1.3 P23：方案设计驱动详解

#### 1.3.1 Requirement→Solution→Spec 链

```
需求输入 → [需求分析] → [架构适配] → [方案设计] → [Spec 生成] → 编码执行
            DP0            DP0.5         DP0.7          DP1
```

| 阶段 | 输入 | 输出 | 验证 |
|------|------|------|------|
| **需求分析** | 用户原始需求 + 业务背景 | 结构化需求文档 | DP0：人工确认 |
| **架构适配** | 结构化需求 + 现有架构文档 | 架构适配分析 | DP0.5：人工确认 |
| **方案设计** | 架构适配分析 + 设计模板 | 方案设计文档 | DP0.7 + Solution Quality Gate |
| **Spec 生成** | 方案设计文档 + Spec 模板 | Spec 文件 | DP1：人工确认 |

#### 1.3.3 需求拆解分析

Spec 生成前，AI 必须对每个用户故事完成需求拆解：

| 维度 | 问题 | 示例 |
|------|------|------|
| **隐性能力** | 需求隐含了哪些系统性能力？ | "查询天气" → 隐含"系统能学习新能力" → 隐含"skill 能泛化" |
| **质量进化** | 产出物如何随时间变好？ | 自动生成的 skill 需渐进式泛化，不是一次性产物 |
| **失败模式** | 如果产出物质量差会怎样？ | 弱泛化 skill 导致意图匹配失败 → 需弱匹配检测 |
| **度量标准** | 如何判断"做好了"而非"做完了"？ | 不是"skill 注册成功"，而是"覆盖 ≥3 种变体意图" |

#### 1.3.4 Solution Quality Gate

方案设计完成、Spec 生成前必须通过 8 项检查：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | **需求覆盖** | 覆盖所有已确认的验收标准 |
| 2 | **架构一致性** | 与现有架构文档无冲突 |
| 3 | **接口完整性** | 所有新增/变更接口有明确输入输出定义 |
| 4 | **数据模型正确** | 有明确的迁移方案 |
| 5 | **异常处理** | 每个关键路径有异常处理策略 |
| 6 | **可测试性** | 每个验收标准有对应测试策略 |
| 7 | **依赖明确** | 新增依赖有版本约束和替换方案 |
| 8 | **风险评估** | 识别 Top 3 技术风险及缓解措施 |

#### 1.3.6 Skill Generalization

需求→Spec 链完成后触发：

1. **模式识别**：提取本次使用的设计/架构模式
2. **跨项目复用**：评估模式是否适用于其他项目
3. **知识沉淀**：将有效模式写入 `domain-knowledge/`
4. **反模式记录**：将失败方案写入 `historical-lessons.md`

### 1.4 自治等级模型（L1-L4）

| 等级 | 名称 | 人类干预点 | 适用场景 |
|------|------|-----------|---------|
| **L1** | 辅助编码 | 每一步 | 新手团队、安全敏感项目 |
| **L2** | 半自主编码 | 每个 PR 合并前 | **日常开发（推荐默认）** |
| **L3** | 受限自主编码 | PR 合并前 + DCP 门禁 | 夜间/周末开发、成熟团队 |
| **L4** | 完全自主编码 | DCP 门禁 + 定期审计 | 高成熟度团队、低风险变更 |

#### 升级路径

| 升级 | 前置条件 | 验证方式 |
|------|---------|---------|
| **L1 → L2** | ≥20 PR 无事故、TDD ≥80%、两层 Review 已建立、CI Gate 已配置 | 检查 PR 历史、.gate/tdd-report.json、最近 5 个 PR 签名 |
| **L2 → L3** | L2 运行 ≥1 月、自主成功率 ≥70%、幻觉率 <5%、异步 DP 机制已建立 | 计算 PR 通过率、验证 Slack 通知/DP 模板/超时策略 |
| **L3 → L4** | L3 运行 ≥3 月、自主成功率 ≥85%、每周审计 ≥95%、零事故、回滚机制已演练 | 检查 4 周审计报告、回滚演练记录、监控告警配置 |

#### 降级条件（满足任一即降级）：

| 触发条件 | 降级目标 |
|---------|---------|
| 生产安全事故（AI 代码导致） | 降到 L2 |
| 幻觉代码合并到 main | 降到 L2 |
| 审计通过率连续 2 周 < 95%（L4） | 降到 L3 |
| 自主成功率连续 2 周 < 70%（L3） | 降到 L2 |
| TDD 执行率 < 80% | 降到 L1 |
| 密钥泄露到代码仓库 | 降到 L1 |

### 1.5 设计启发式（AI 可执行的架构思考）

规范不仅定义"什么不能错"，还定义"如何确保做对"。AI 在架构/设计阶段必须应用以下思考框架：

#### 1.5.1 数据流分析法

对于每个功能，追踪一条完整数据流：

1. 数据从哪里产生？（用户输入 / 文件 / DB）
2. 经过哪些转换？（解析 / 校验 / 序列化）
3. 最终到哪里？（DB / 响应 / 日志）
4. 每一步的类型是否一致？（JSONB vs TEXT[] vs string）

#### 1.5.2 生命周期分析法

对于每个创建的实体，追问：

1. 创建后如何被读取？
2. 如何被更新？（谁触发、触发条件）
3. 如何被删除/归档？
4. 如果从未被更新，会怎样？

#### 1.5.3 时间维度分析法

对于每个系统能力，追问：

1. 第一次使用时是什么样？
2. 使用 10 次后是什么样？
3. 使用 1000 次后是什么样？
4. 系统是变好了（学习）还是变差了（熵增）？

#### 1.5.4 代码构造检查清单

每次写入代码前，AI 必须执行以下正向检查（与现有 A01-A09 的区别：A01-A09 是"审查时检查问题"，这是"写代码前检查动作"）：

| 类别 | 检查项 |
|------|--------|
| **数据持久化** | SQL INSERT/UPDATE 包含目标表所有非空列；JSONB 列使用 `json.Marshal`，TEXT[] 列使用 Go slice；新增字段确认 migration 已存在 |
| **字符串处理** | `fmt.Sprintf` 中的字面量 `%` 使用 `%%` 转义；用户输入经 sanitization 后再拼入 SQL/log/URL |
| **并发安全** | 所有读写 map 的操作有 mutex 保护；goroutine 中的闭包变量已拷贝 |
| **输入校验** | 所有外部输入经过类型校验 + 边界检查；路径参数经清理防止路径穿越 |
| **错误处理** | 所有错误被返回或处理；禁止 `//nolint:errcheck`；错误消息不泄露内部细节 |

### 1.6 IPD 流程与方法引擎

AI Coding 不是从编码开始的——70% 的失败源于 Phase 0-1 的方向错误。每个阶段必须有明确的方法、AI 辅助方式和决策标准。

> **核心认识**：IPD 的 DCP（Decision Check Point）是"要不要做"，本规范关注"用什么方法确保决策质量"。

#### 1.6.1 Phase 0：市场洞察

**目标**：明确"为什么要做"——验证商业机会的真实性。

| 方法 | 说明 | AI 辅助方式 | 产出物 |
|------|------|-----------|--------|
| **五看三定** | 看行业/市场/客户/竞争/自己 → 定目标/策略/路径 | AI 自动采集竞品功能、定价、技术栈；生成趋势分析报告 | 市场洞察报告 |
| **竞品范围确认** | 在竞品分析前，AI 必须生成竞品范围声明，覆盖≥3 个不同维度类别（直接竞品/间接竞品/替代方案/学术前沿/平台生态），每个维度至少 1 个具体代表 | AI 根据领域自动生成分类框架；CI 自动检查维度数量≥3 且来源多样性≥2，否则阻断 | 竞品范围声明 |
| **BLM 模型** | 战略意图→市场洞察→创新焦点→业务设计 | AI 生成多套业务设计方案的优劣势对比 | 业务设计草案 |
| **技术趋势雷达** | Gartner Hype Cycle 评估技术成熟度 | AI 抓取 GitHub/论文/博客趋势，标注技术生命周期 | 技术选型建议 |
| **VOC 分析** | 客户之声：从用户反馈中提取真实需求 | AI 对工单/评论/访谈文本做情感分析和主题聚类 | 客户需求优先级矩阵 |

**DCP 检查**：商业机会是否经过五看验证？是否排除了"伪需求"？是否有明确的差异化定位？**竞品范围是否覆盖≥3 个维度类别且至少有 1 个非直接竞品**？

#### 1.6.2 Phase 1：概念定义

**目标**：明确"做什么"——将市场需求转化为可执行的产品定义。

| 方法 | 说明 | AI 辅助方式 | 产出物 |
|------|------|-----------|--------|
| **$APPEALS** | 8 维度竞争力评估：价格/可获得性/包装/性能/易用性/保证/生命周期成本/社会接受度 | AI 自动对标竞品在各维度的得分，生成差距分析 | 竞争力差距矩阵 |
| **Kano 模型** | 基本型 vs 期望型 vs 兴奋型需求分类 | AI 根据用户反馈频率和满意度预测自动分类；**兴奋型需求必须有前瞻性来源标注（学术论文/新兴产品/替代方案），来源类型字段不得为"现有直接竞品功能"** | 需求优先级排序 |
| **QFD** | 质量功能展开：客户需求→技术参数的映射矩阵 | AI 自动将用户需求映射到技术规格，标记冲突项 | 技术需求规格书 |
| **Jobs-to-be-Done** | 客户"雇佣"产品完成什么工作？ | AI 从用户访谈中提取 JTBD 语句模板："我想在{场景}下{做什么}，以便{达成什么}" | JTBD 需求清单 |
| **价值曲线** | 可视化产品在各竞争要素上的表现 | AI 根据竞品数据自动生成价值曲线对比图 | 价值曲线图 |

**DCP 检查**：需求分类是否覆盖 Kano 三类型？QFD 矩阵是否完整？JTBD 是否覆盖了核心用户场景？

#### 1.6.3 Phase 2：详细规划

**目标**：明确"怎么做"——将产品定义转化为可执行的开发计划。

| 方法 | 说明 | AI 辅助方式 | 产出物 |
|------|------|-----------|--------|
| **DFX** | Design for X：可维护性/可扩展性/安全性/性能 | AI 自动执行 DFX 检查清单，对照架构文档逐项评分 | DFX 评估报告 |
| **架构权衡分析(ATA)** | 评估架构方案对各质量属性的影响 | AI 生成 2-3 套候选架构的权衡对比表 | 架构决策记录(ADR) |
| **WBS 分解** | 工作包分解到 Feature 粒度 | AI 从 PRD 自动生成 WBS 草案，标注依赖关系和估算 | WBS + 依赖图 |
| **风险矩阵** | 概率×影响，Top 风险识别和缓解 | AI 根据历史项目数据自动标注风险概率 | 风险登记册 |
| **里程碑规划** | 关键路径识别和交付节点 | AI 基于 WBS 和依赖关系生成甘特图和关键路径 | 项目计划 |

**DCP 检查**：DFX 是否全部通过？架构方案是否经过权衡分析？WBS 是否分解到 Feature 粒度？Top 3 风险是否有缓解方案？

#### 1.6.4 Phase 3：AI 辅助开发

**目标**：高质量交付——用 AI 最大化编码效率和质量。

本阶段是 v5.4 规范的核心覆盖区域，详见全文各章节。关键方法：

| 方法 | 说明 | 对应章节 |
|------|------|---------|
| **TDD 先行(P3)** | 测试→实现→重构，Red→Green→Refactor | 第 2 章 |
| **Spec 驱动(P7)** | 需求→Spec 链，编码前完成方案设计 | 第 4 章、1.3 节 |
| **最小批量(P8)** | 一次一个函数/模块，>50 行必须拆分 | 1.1 节 |
| **Self-Correction Loop** | 自动修复最多 3 轮 | 2.2 节 |
| **两层审查(P4)** | AI Reviewer 检测幻觉 + Human Reviewer 专注业务逻辑 | 3.2 节 |
| **质量降级机制** | AI 输出质量持续不达标时自动降级 | 2.5 节 |
| **设计启发式** | 数据流分析/生命周期分析/时间维度分析 | 1.5 节 |
| **构造检查清单** | 数据持久化/字符串处理/并发安全/输入校验/错误处理 | 1.5.4 节 |

**DCP 检查**：TDD 合规率 ≥ 80%？幻觉率 < 5%？Spec 覆盖度 100%？所有 DFX 约束已满足？

#### 1.6.5 Phase 4：验证发布

**目标**：确保"做好了"——在真实用户场景下验证质量。

| 方法 | 说明 | AI 辅助方式 | 产出物 |
|------|------|-----------|--------|
| **E2E 测试** | 端到端真实场景验证 | AI 自动生成 E2E 测试用例，覆盖关键用户旅程 | E2E 测试报告 |
| **Beta 测试管理** | 真实用户场景下的功能和体验验证 | AI 自动收集和分析 Beta 用户反馈 | Beta 测试报告 |
| **GRTR** | 通用技术评审：性能/安全/兼容性 | AI 自动执行 GRTR 检查清单 | GRTR 评审报告 |
| **ADCP** | 可用性决策检查点：功能/质量/文档/支持就绪 | AI 自动汇总发布 Readiness 状态 | ADCP 决策报告 |
| **发布 Readiness** | 功能完整性、质量指标、文档、培训、支持 | AI 自动生成发布检查清单并逐项验证 | 发布 Readiness 清单 |

**DCP 检查**：E2E 测试 100% 通过？Beta 用户满意度达标？GRTR/ADCP 全部通过？发布文档完整？

**目标**：持续改进——从市场反馈到产品迭代的闭环。

|------|------|-----------|--------|
| **客户反馈闭环** | 市场问题→改进需求的端到端闭环 | AI 自动分类和优先级排序客户反馈，关联到 Spec | 改进需求清单 |
| **技术债管理** | 量化技术债、优先级排序、偿还计划 | AI 自动扫描技术债（代码/架构/测试），生成偿还建议 | 技术债登记册 |

**IPD 六阶段与自治等级的关系**：自治等级（L1-L4）仅影响 **Phase 3** 的执行方式。Phase 0-2 和 Phase 4-5 的决策质量由上述方法论保障，不受自治等级影响。

### 1.6.7 阶段间变更传导规则（Phase Change Propagation）

> **核心原则**：IPD 六阶段不是顺序流水线，而是依赖链。下游阶段的决策以上游阶段的产出为输入。

当任一 Phase N 的产出物发生**实质性变化**时（新增、修改、废弃关键假设或结论），必须执行传导检查：

| 步骤 | 动作 | 产出 |
|------|------|------|
| **1. 影响识别** | 列出 Phase N+1 的核心假设，逐一判断是否仍成立 | 影响清单 |
| **2. 局部刷新** | 不成立的假设所在阶段，重新生成受影响章节 | 刷新后的产出物 |
| **3. 逐阶段传导** | 从 Phase N+1 传导至 Phase 5，每阶段输出 PASS 或 REFRESHED | 传导矩阵 |
| **4. 记录** | 写入 `ipd/phase-N/impact-log-{date}.md` | 审计追溯 |

**传导方向与依赖关系**：

| 下游阶段 | 依赖的上游产出 | 上游变化时的检查重点 |
|---------|--------------|-------------------|
| Phase 1 → Phase 0 | 市场洞察报告、竞品范围声明 | 新竞品是否改变差异化定位、优先级排序 |
| Phase 2 → Phase 1 | 概念定义、JTBD、Kano 分类 | 需求优先级变化是否影响架构预留 |
| Phase 3 → Phase 2 | 架构决策、WBS、风险矩阵 | Sprint 范围是否覆盖已识别风险 |
| Phase 4 → Phase 3 | 已交付代码、测试结果 | 发布标准是否因交付质量调整 |
| Phase 5 → Phase 4 | 发布版本、用户反馈 | 生命周期策略是否因市场反馈变化 |

**不触发传导的变化**：格式调整、错别字、不影响结论的措辞修改、补充已有结论的证据。

**触发传导的变化**：新增竞品、差异化定位改变、需求优先级重排、架构决策变更、风险矩阵更新、Sprint 范围调整。

## 第 2 章：TDD 与质量保障

#### Auto-Coding TDD 流程（L2-L4）

```
1. AI 读取 Spec（specs/F{NNN}-{name}.md）
2. 从 Spec 的验收标准（Gherkin）生成测试代码
3. 提交测试代码 → CI 记录 Red 状态
4. 生成实现代码
5. 运行测试 → 必须全部通过（Green）
6. 失败 → Self-Correction Loop（最多 3 轮）
7. 重构 → 测试仍通过
8. 全量验证 Gate
9. 创建 PR → AI Reviewer + Human Reviewer
10. 合并
```

**AI 生成测试特殊规则**：

| 规则 | 说明 | 检查方式 |
|------|------|---------|
| 从 Spec 生成 | 测试必须基于 Spec 中的验收标准 | 对照 Spec AC 条目 |
| 先于实现 | 测试必须在实现之前提交 | git log 提交顺序 |
| 审核断言 | AI 倾向于生成"能通过"而非"正确"的断言 | L1/L2：人工审核每个断言 |
| 覆盖异常路径 | AI 倾向只生成正常路径测试 | CI 检查分支覆盖率 |
| 避免过度 Mock | AI 倾向为一切写 Mock | 人工审查 Mock 必要性 |

#### TDD 合规检查 + 造假检测

CI 必须自动检测以下 TDD 合规项，任何一项不通过即阻塞合并：

| 检查项 | 检测方法 | 不通过后果 |
|--------|---------|-----------|
| 提交顺序 | `_test` 文件的 commit 必须在实现 commit 之前 | 标记"TDD 未遵循" |
| Red 阶段 | 测试提交后、实现提交前，测试必须失败 | 标记"TDD Red 阶段缺失" |
| Green 转换 | 实现提交后，测试必须通过 | 阻塞合并 |
| AC 覆盖率 | 每个 Spec 的验收标准 100% 有对应测试+证据 | 阻塞合并 |
| 边界覆盖 | 每个 AC 至少覆盖正常路径 + 2 种异常路径 | 阻塞合并 |
| 包覆盖率 | 趋势监控，不得较基线下降 > 5% | 警告 |
| 新增代码覆盖率 | 新增代码（新函数/新文件）必须有测试 | 阻塞合并 |
| 存量代码覆盖率 | 作为长期目标，按 Sprint 优先级渐进提升 | 不阻塞日常合并 |
| 连续违规 | 同一 PR 中存在 2 项或以上 TDD 违规（如提交顺序错误 + Red 阶段缺失） | Gate 不通过 |

**TDD 造假检测**：AI 可能在一次提交中同时提交测试和实现，使流程看起来被执行但实际没有。CI 必须检查提交粒度（测试和实现不得在同一 commit 中）、验证 Red 状态（CI 记录测试失败）、记录转换证据到 `.gate/tdd-report.json`。连续 2 次违规 = 触发警告并写入周报。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Red 阶段    │────▶│ Green 阶段   │────▶│ Refactor 阶段│
│  写测试      │     │ 写实现       │     │ 重构代码     │
│  运行 → 失败 │     │ 运行 → 通过  │     │ 运行 → 通过  │
└─────────────┘     └─────────────┘     └─────────────┘
```

| 阶段 | 要求 |
|------|------|
| **Red** | 测试文件必须先提交且运行失败，CI 记录 Red 状态 |
| **Green** | 实现提交后所有测试通过，CI 验证 Red→Green 转换 |
| **Refactor** | 重构后测试仍全部通过，CI 验证无回归 |

### 2.2 Self-Correction Loop（最多 3 轮）

> **注**：各 Pipeline 层级的自修轮次限制可能不同，详见 06-cicd-pipeline.md 第 4.1 节。通用 Self-Correction Loop 的 3 轮限制适用于 L2 测试层和 L4 集成层。

```
AI 生成代码 → 自动化检查 → 通过→Code Review
                         → 失败→Round 1→通过→Code Review
                                    →失败→Round 2→通过→Code Review
                                               →失败→Round 3→通过→标记"3 轮后通过"
                                                          →失败→停止，通知人工
```

**适用场景**：Lint 错误（80%）、类型错误（85%）、测试失败（70%）、API 幻觉（60%）
**不适合**：架构不合理（20% 成功率）

### 2.3 全量验证 Gate

```
1. 编译/类型检查
2. 全量测试
3. lint
4. 失败 → Self-Correction Loop（最多 3 轮）
5. 3 轮后仍未通过 → 停止，向人类报告
```

**覆盖率规则**：
- **AC 覆盖率**（强制）：每个 Spec 验收标准必须有≥1 测试函数 + ≥2 证据项，AC 覆盖率<100%=未完成
- **新增代码**（强制）：新函数/新文件必须有对应测试，否则阻塞合并
- **包覆盖率**（趋势防护）：整体包覆盖率不得较基线下降 > 5%，作为 CI 警告门禁
- **80% 行覆盖率目标**：作为长期质量目标，不作为 CI 硬性阻断门禁（AI 时代下该指标可通过 AI 自动生成测试快速达成，但人工审查成本高，应优先保证 AC 覆盖而非数字达标）
- **证据文件**：写入 `.gate/coverage.json`

### 2.4 AI-First-Then-Human-Refine 模式

TDD 先行的补充，仅适用于：快速原型验证、探索性编码、UI/UX 原型。规则：必须在 PR 中说明原因，合并前仍须补齐测试，同一模块连续使用超过 3 次需启动根因分析。

### 2.5 质量降级与回退机制

当 AI 输出质量持续不达标时，自动降级开发模式：

| 级别 | 触发条件 | 动作 | 恢复条件 |
|------|---------|------|---------|
| **Level 0 正常** | AI 代码一次通过率 > 80% | AI 生成，人工审核 | — |
| **Level 1 警惕** | AI 代码一次通过率 60-80% | 加强审核，记录问题模式 | 连续 3 个 Feature 通过率 > 80% |
| **Level 2 降级** | AI 代码一次通过率 < 60% | 改为"人写框架，AI 填充细节" | 连续 5 个 Feature 通过率 > 70% |
| **Level 3 回退** | AI 提供者质量严重退化 | 人工编码，AI 仅辅助（注释、lint） | AI 提供者问题解决 |

**注意**：质量降级不同于自治等级降级。质量降级影响 AI 生成代码的信任级别，自治等级降级影响整个开发流程的自主程度。

## 第 3 章：AI 幻觉检测与防护

### 3.1 核心认识

AI 幻觉是 LLM 的本质特征，**每次生成都可能发生**。Gate 的存在不是为了检测是否发生，而是确保即使发生也不会进入生产。

### 3.2 防护策略

| 策略 | 说明 |
|------|------|
| **Example-Driven Prompting** | 用 `.examples/` 中的代码示例替代规则描述，AI 对示例的理解是规则的 5 倍。错误做法："请确保代码遵循单一职责原则"；正确做法："请参考 `.examples/architecture/repository-pattern.md` 的函数拆分方式，按相同风格编写"。示例必须来自真实项目代码，包含问题描述、错误示例、正确示例、为什么这样做好 |
| **Prompt Chaining** | 将复杂任务分解为顺序链：分析→设计→编码→验证，每步验证输出质量。链配置：`prompt_chaining.max_retries_per_step=3`，失败转人工。示例：分析（列出需修改函数）→设计（修改方案）→编码（实现）→验证（对照方案检查）→修复（处理问题） |
| **Progressive Disclosure** | 按需加载上下文，通过优先级队列和动态替换保持高注意力。P0（Spec、接口契约）每次对话开始加载；P1（相关源文件、架构约束）首次提到时加载；P2（商业计划、项目历史）手动注入。每个 Spec 必须声明 `required_context` 精确列出需要的文件和行范围。对话中不再相关的 P1 上下文应被移除；超出上下文窗口时拆分任务 |
| **AI Reviewer + Human Reviewer** | 两层审查，AI Reviewer 检测幻觉，Human Reviewer 专注业务逻辑 |

> **前置链**：Prompt Chaining 用于编码执行阶段，必须先完成 P23 的 Requirement→Solution→Spec 链（编码前）。

### 3.3 检测方法

| 方法 | 检测的幻觉类型 |
|------|---------------|
| 编译/类型检查 | API 幻觉、类型不匹配 |
| 符号解析 | 未定义的符号引用 |
| 依赖验证 | 依赖幻觉、版本幻觉、Typosquatting |
| SAST 扫描 | 安全幻觉、漏洞 |
| AC 映射验证 | 覆盖率幻觉、完成幻觉 |

### 3.4 指标

| 指标 | 目标 | 告警阈值 |
|------|:----:|---------|
| 幻觉发生率 | < 5% | > 10% |
| 幻觉拦截率 | 100% | < 90% |
| 幻觉逃逸率 | 0% | > 0% |

## 第 4 章：Spec 驱动开发

### 4.1 Feature Spec 规范

每个 Feature 必须有且仅有一个 Spec 文件，存放在 `specs/` 目录：

```yaml
---
type: feature-spec
id: F001
name: "用户注册"
version: 1.0
status: draft | validated | ready | in-progress | done | deprecated
priority: must | should | could | wont
validated_by: "@name"
required_context:
  - file: "docs/architecture.md", lines: "50-80"
required_domain_knowledge:
  - domain-knowledge/industry/{domain}.md
  - domain-knowledge/tech-stack/{stack}.md
autonomy_level: L2
---
```

Spec 必须包含以下内容（不必按此顺序，每项均为必须，不可省略）：用户故事、验收标准（Gherkin 格式）、边界条件、数据模型、API 设计、非功能需求。

### 4.2 Spec Validation Gate

| 验证项 | 检查要点 |
|--------|---------|
| 与 PRD 一致性 | 每个 PRD 需求至少有一个 AC 对应 |
| 与架构一致性 | 不违反任何 ADR |
| 验收标准可测试 | Gherkin 格式可执行 |
| 边界条件完整 | 至少覆盖正常路径 + 3 种异常路径 |
| 非功能需求明确 | 数值化，不可用模糊描述 |

### 4.3 Spec 状态机

```
draft →(人工审核)→ validated →(放入队列)→ ready →(AI 开发)→ in-progress →(Gate+Review)→ done
                                                                 any →(取消)→ deprecated
```

## 第 5 章：Prompt 管理

### 5.1 Prompt 版本化（P9）

每个用于生成代码的 Prompt 必须存储在 `prompts/` 目录，包含 YAML frontmatter（id、version、model、temperature、feature、status）。同一 Prompt 每次修改递增版本号。Prompt 一次通过率 < 50% = 需要重写。

### 5.2 Prompt 安全

- Prompt 文件必须来自本地仓库，禁止从 URL 或外部服务加载
- Spec 中的用户输入不得直接拼接到 Prompt 中
- 每季度审查 Prompt 仓库，删除过时 Prompt

## 第 6 章：人与 AI 的分工

### 6.1 基础分工

| 活动 | 人类角色 | AI 角色 |
|------|---------|---------|
| 战略/产品/架构决策 | **决策者** | 信息整理、方案对比 |
| Spec 编写 | 审核者 | 草稿生成（L2+） |
| 测试编写/编码实现 | 审核者 | **执行者** |
| 安全审查 | **决策者** | 辅助扫描 |
| 部署运维 | 监督者（L1-L2）/ 审计者（L4） | **执行者** |

### 6.2 Decision Point

| # | 决策点 | 时机 | 拦截的错误类型 |
|---|--------|------|---------------|
| **DP0** | 需求确认 | 需求分析完成后 | 需求理解偏差 |
| **DP0.5** | 架构方向 | 架构适配完成后 | 架构不兼容 |
| **DP0.7** | 方案可行性 | 方案设计完成后 | 方案不可行 |
| **DP1** | 需求理解 | AI 分析完代码结构后 | AI 误解需求 |
| **DP2** | 架构方案 | AI 给出技术方案后 | 过度工程 |
| **DP3** | 发布决策 | 全部完成后 | 功能不完整 |
| **DP4** | 紧急变更 | Hotfix 时 | 修复引入新问题 |

**L3 异步模式**：DP1/DP2 阻塞，DP3 非阻塞，超时 2 小时自动降级为非阻塞。

### 6.3 DCP 与 Decision Point 的关系

| | DCP | Decision Point |
|---|---|---|
| 粒度 | 阶段级 | 任务级 |
| 频率 | 每个 Phase 一次 | 每个任务多次 |
| 关系 | 宏观保护 | 微观保护 |

两者互补，不可替代。
