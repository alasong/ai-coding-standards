## 第 1 章：规范生命周期

### 1.1 状态机

```
draft → review → active → deprecated → archived
```

| 状态 | 定义 | 可执行操作 |
|------|------|-----------|
| **draft** | 提案阶段，内容未定稿 | 任意编辑、讨论 |
| **review** | 提交正式评审 | 仅可添加评论，不可改正文 |
| **active** | 生效中，AI 必须遵守 | 仅可通过 SCR 流程修改 |
| **deprecated** | 已废弃，有替代方案 | 只读，保留参考 |
| **archived** | 历史归档 | 只读，不可引用 |

### 1.2 状态转换条件

| 转换 | 触发条件 | 审批方 |
|------|---------|--------|
| draft → review | 提案完成，填写 SCR（见第 2 章）+ Gate Checker Agent 验证 SCR 完整性 | 提案人提交 → Gate Checker 预检 → 审批委员会 |
| review → active | 评审通过（见 1.3 审批矩阵） | 审批委员会 |
| active → deprecated | 被新规范替代 | 审批委员会 |
| deprecated → archived | 保留期满（默认 90 天） | 自动 |

### 1.3 审批矩阵

| 变更类型 | 定义 | 提案人 | 评审人 | 批准人 |
|---------|------|--------|--------|--------|
| **Editorial** | 错别字、格式、链接修复 | 任何人 | — | 自动合并 |
| **Minor** | 补充说明、示例、不影响规则语义 | 任何人 | 1 名 Maintainer | 1 名 Maintainer |
| **Major** | 新增/修改/删除规则，影响 AI 行为 | Maintainer | ≥2 名 Maintainer + 1 名领域专家 | ≥2/3 Maintainer 同意 |
| **Breaking** | 不兼容变更，AI 需重新训练/配置 | Maintainer | 全体 Maintainer + 技术负责人 | 技术负责人 + ≥3/4 Maintainer |

**Maintainer**：拥有规范仓库 write 权限的人员，初始 3-5 人。

## 第 2 章：规范变更请求（SCR）

### 2.1 SCR 模板

每个 Major/Breaking 变更必须提交 SCR：

```markdown
# SCR-{编号}：{简短标题}

## 元信息
- **类型**：Major / Breaking
- **影响规范**：[文件名列表]
- **提案人**：{姓名}
- **日期**：{YYYY-MM-DD}

## 问题陈述
{当前规范哪里不足，用事实或数据说明}

## 变更内容
{具体改了什么，附 diff 或新章节}

## 影响分析
- AI 行为变化：{改前 vs 改后}
- 存量 PR 影响：{已生成代码是否需要调整}
- 工具链影响：{CI/Gate/脚本是否需同步}
- 迁移成本：{低/中/高}

## 替代方案
{考虑过的其他方案及排除理由}

## 过渡计划
{见第 3 章过渡期规则}
```

### 2.2 Editorial/Minor 变更

不需 SCR，直接 PR。PR 描述中注明变更类型和影响范围。

### 2.3 紧急修复通道

规范存在明显错误（如规则矛盾、引用断裂）时：
1. Maintainer 可直接修复并标注 `[HOTFIX]`
2. 48 小时内补齐正式 SCR（追溯评审）
3. 仅适用于 Editorial/Minor 级别

## 第 3 章：过渡期规则

### 3.1 过渡期定义

| 变更类型 | 过渡期 | 规则 |
|---------|--------|------|
| **Major** | 30 天 | 新旧规则均可，过渡期后仅接受新规则 |
| **Breaking** | 60 天 | 旧规则标记 deprecated，过渡期后自动 archived |

### 3.2 过渡期执行

```
生效日 ───────────── 过渡期结束 ──→
│                      │              │
新规范 active          │              旧规范 archived
旧规范 deprecated ─────┘              新规范唯一有效
```

- 过渡期内创建的 PR 可按旧规范执行，但必须在 PR 中标注 `[LEGACY-SPEC]`
- AI 在过渡期内必须优先尝试新规范，旧规范仅在不适配时回退
- 过渡期结束前 7 天，CI 自动扫描仍有 `[LEGACY-SPEC]` 标记的 PR 并告警

### 3.3 向后兼容原则

规范版本遵循 [14-release-management.md](14-release-management.md) SemVer 规则：
- **Patch**：编辑性修正，完全兼容
- **Minor**：新增内容，向后兼容
- **Major**：不兼容变更，必须走过渡期

## 第 4 章：规范版本管理

### 4.1 版本标注

每个规范文件头部标注 `版本：vX.Y | YYYY-MM-DD`。
规范集整体版本由 `INDEX.md` 统一声明。

### 4.2 规范间依赖

| 依赖类型 | 处理方式 |
|---------|---------|
| **前置依赖** | 被依赖规范未 active，依赖方不得进入 active |
| **交叉引用** | 引用断裂 = CI 阻断（见 4.4） |
| **规则冲突** | 两条规范矛盾 = `[SPEC-CONFLICT]`，提交 SCR 解决 |

### 4.3 规范目录结构

```
ai-coding-v5.4/
├── INDEX.md                    # 总索引 + 整体版本
├── 01-core-specification.md    # 核心规范（不可删除）
├── 02-18-*.md                  # 领域规范
├── 18-spec-evolution-governance.md  # 本文件
└── deprecated/                 # 已废弃规范归档
    └── {原文件名}
```

### 4.4 CI 验证

CI 必须检查：
- [ ] 所有交叉引用指向有效文件
- [ ] 无 `[SPEC-CONFLICT]` 未解决标记
- [ ] 版本号格式正确
- [ ] 无 orphan 规则（引用了不存在的规范章节）

## 第 5 章：反馈与改进机制

### 5.1 反馈渠道

| 来源 | 处理方式 |
|------|---------|
| AI 运行时检测矛盾 | 自动创建 Issue，标记 `spec-feedback` |
| 人工审查发现不足 | PR 评论或直接创建 SCR |
| 事故复盘结论 | 事故报告 → SCR → 规范修正 |
| 定期审查 | 每季度 Maintainer 审查全部规范 |

### 5.2 定期审查

| 频率 | 内容 | 参与方 |
|------|------|--------|
| **月度** | 检查规范引用有效性、过期过渡期 | 自动化 CI |
| **季度** | 审查规则合理性、删除过时内容 | Maintainer |
| **半年** | 全面评估规范集是否覆盖当前实践 | Maintainer + 技术负责人 |

### 5.3 规范度量指标

| 指标 | 目标 | 用途 |
|------|------|------|
| 规范违反率 | < 5% PR | 评估规范可执行性 |
| SCR 处理周期 | < 14 天 | 评估治理效率 |
| 规则矛盾数 | 0 | 评估规范一致性 |
| 规范覆盖率 | > 90% 场景 | 评估规范完整性 |

## 第 6 章：AI 在规范演进中的角色

### 6.1 AI 可执行

| 操作 | 说明 |
|------|------|
| 检测规范矛盾 | 运行时发现两条规则冲突，报告 `[SPEC-CONFLICT]` |
| 检测引用断裂 | 引用章节不存在，报告 `[BROKEN-REF]` |
| 生成 SCR 草稿 | 基于 Issue/事故报告自动生成 SCR 模板内容 |
| 统计指标 | 自动计算规范违反率、覆盖率等 |

### 6.2 AI 不可执行

| 操作 | 原因 |
|------|------|
| 批准规范变更 | 需要人类判断影响 |
| 修改 active 状态的规范 | 必须走 SCR 流程 |
| 删除任何规范文件 | 必须经 deprecated → archived 流程 |
