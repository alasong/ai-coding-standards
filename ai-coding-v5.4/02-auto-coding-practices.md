## 第 1 章：Auto-Coding 总览

| 维度 | AI 辅助编码 | Auto-Coding |
|------|------------|-------------|
| 执行模式 | 人+AI 逐行补全 | AI 独立完整开发循环 |
| 干预频率 | 每次编辑参与 | 仅里程碑/审查点介入 |
| 适用等级 | L1 | L2-L4 |

各等级能力：

| 能力 | L1 | L2 | L3 | L4 |
|------|----|----|----|----|
| 独立 TDD | 否 | 是 | 是 | 是 |
| 自修 3 轮 | 人工确认 | 自动 | 自动 | 自动 |
| specs/ 自动取任务 | 否 | 否 | 是 | 是 |
| 夜间/周末运行 | 否 | 否 | 是 | 是 |
| 自动合并 | 禁止 | 禁止 | 禁止 | 仅 trivial |

**最高 ROI**：依赖更新、CI 自修复、夜间特性、机械重构、测试生成
**最高风险**：安全代码、DB 迁移、架构重构、性能关键路径

## 第 2 章：自主编码模式（引用）

本章涉及的核心机制定义在 [01-core-specification.md](01-core-specification.md) 中：

| 概念 | 定义位置 | 说明 |
|------|---------|------|
| TDD 循环 | 01-core §2 | Red → Green → Refactor，提交顺序验证，Red 阶段记录 |

**修复禁忌**（02-auto-coding 独有）：不得修改测试断言使其"通过"、不得删除失败测试、不得添加 `@skip`、不得吞异常。

## 第 3 章：定时任务与持续执行

### 3.1 三种定时任务

| 模式 | 环境 | 关机运行 | 场景 |
|------|------|---------|------|
| Cloud Scheduled | 云端 | 是 | 依赖更新、文档生成 |
| Desktop Scheduled | Desktop | 否 | 构建、测试、Docker |
| CLI /loop | 会话 | 否 | 轮询、监控 |

### 3.2 Kill Switch

| 级别 | 触发 | 动作 |
|------|------|------|
| L1 警告 | >4h 或 >$10 | 通知不中断 |
| L2 停止 | >8h 或 >$30 | 中断通知 |
| L3 紧急 | 安全事件/泄露 | 中断+回滚+告警 |

### 3.3 异步 Decision Point（L3/L4）

```
AI 到达 DP → 生成摘要 → 发送人工审核者
  ├── 阻塞：DP1（需求理解）、DP2（架构方案）
  ├── 非阻塞：DP3（发布），人工不认可则 revert
  └── 超时 2h 未确认 → 降级非阻塞 + 警告
```

规则：DP1/DP2 必须阻塞；人工确认必须具体，禁用"OK"等模糊回复。

### 3.4 夜间开发前置检查

- [ ] Spec ready（specs/ 中 status=ready）
- [ ] Kill Switch 已设置
- [ ] 通知 Channel 已测试
- [ ] AC 覆盖率 100%

## 第 4 章：自修复 CI

### 4.1 自动修复范围

| 类别 | 修复 | 示例 |
|------|------|------|
| 格式/命名/导入 | 是 | 缩进、未使用导入 |
| 类型错误 | 最小修改 | 缺少类型标注 |
| Lint 警告 | 是 | 代码风格 |
| 测试失败 | ≤3 轮 | Mock 配置错误 |

转人工：安全漏洞、架构问题、DB 迁移、性能问题、业务逻辑错误、>3 轮自修、>50 文件、密钥配置。

### 4.2 Flaky Test

条件：同一测试连续 3 次 CI 通过→失败→通过，或失败消息含时序关键词。处理：记录到 `.flaky-tests.yml`，标记不阻塞。

## 第 5 章：Supervisor-Worker 编排

| 角色 | 模型 | 职责 |
|------|------|------|
| Supervisor | Opus | 需求分析、任务拆分、分发、仲裁 |
| Worker | Sonnet | TDD 循环执行 |

流程：读取 ready Spec → 构建依赖图 → 拆分原子任务 → 按依赖分发 → 并行执行无依赖任务 → 汇总 PR → 全量测试 + 幻觉扫描 + 安全检查

## 第 6 章：Auto-Coding 度量

| 指标 | 计算 | 目标 |
|------|------|:----:|
| 自主成功率 | 一次通过审查 PR / 总 PR | ≥70% (L3) / ≥85% (L4) |
| 幻觉率 | 含幻觉 PR / 总 PR | <5% |
| TDD 合规率 | TDD 流程 PR / 总 PR | ≥80% |
| 单次 PR 成本 | 总 API 成本 / PR 数 | 下降趋势 |

预警：自主成功率 <70% 连续 2 周→降级；幻觉率 >10%→暂停；TDD <80%→L1；密钥泄露→停机回滚。

## 第 7 章：IPD 轻量化执行指南（单人项目）

> 原则：方法论完整保留，执行成本降至 10%。

### 7.1 幻觉检测：只自动执行 5 种高频类型

| 类型 | 检查 | 验证 |
|------|------|------|
| E01 不存在 API | import 的包/函数存在 | 编译 |
| X01 操作缺失 | 声明的操作已执行 | 运行 |
| V01 测试伪造 | 测试真实执行断言 | 覆盖率 |
| L01 条件反转 | 逻辑条件正确 | 审查 |
| S01 参数化 | SQL/命令无注入 | SAST |

其余 40 种类型作为 PR 审查参考清单，不自动扫描。

### 7.2 DP 门禁：嵌入 PR 审查，不额外建文档

| DP | 时机 | 检查 | 产出 |
|----|------|------|------|
| DP0 | Spec PR 合并 | 用户故事 + 验收标准 | PR 描述声明 |
| DP0.5 | 架构变更 PR | 不违反 ADR | PR 描述引用 ADR |
| DP0.7 | Solution PR | 8 项 Quality Gate | PR 描述声明 |
| DP1 | 编码 PR | 读取 Spec，状态正确 | PR 描述声明 Spec 路径 |
| DP3 | PR 合并前 | 测试全绿 + lint 全过 | CI 输出 |

### 7.3 证据链：PR 描述即证据聚合

PR 描述必须包含：
```
## Evidence Chain
- 意图：Spec 路径
- 计划：TDD Red→Green
- 行动：diff link
- 验证：CI 测试全绿 + go vet 零问题
- 审查：AI Reviewer + Human Reviewer
```

交叉引用定义：git log + CI 输出 + Spec 文件 = 3 条独立证据。

### 7.4 风险矩阵：Sprint 回顾 5 分钟

| 动作 | 频率 |
|------|------|
| 风险状态检查 | 每 Sprint |
| 热力图更新 | 季度 |
| 新增风险 | 新功能/架构变更时 |

### 7.5 阶段传导：实质变化才执行

| 触发 | 动作 |
|------|------|
| 竞品/差异化/架构决策变更 | 执行传导 |
| 格式/措辞/补充证据 | 跳过 |

记录格式：变化源 → 影响清单 → PASS/REFRESHED。
