## 第 1 章：Auto-Coding 场景总览

### 1.1 什么是 Auto-Coding

| 维度 | AI 辅助编码 | Auto-Coding |
|------|------------|-------------|
| 执行模式 | 人+AI 逐行补全 | AI 独立执行完整开发循环 |
| 干预频率 | 每次编辑都需要人参与 | 仅在里程碑/审查点介入 |
| 适用自治等级 | L1 | L2-L4 |

### 1.2 各等级能力

| 能力 | L1 | L2 | L3 | L4 |
|------|----|----|----|----|
| 独立 TDD 循环 | 否 | 是 | 是 | 是 |
| 自修循环（3 轮） | 人工确认每轮 | 自动执行 | 自动执行 | 自动执行 |
| 从 specs/ 自动获取任务 | 否 | 否 | 是 | 是 |
| 夜间/周末持续运行 | 否 | 否 | 是 | 是 |
| 自动合并到 main | 禁止 | 禁止 | 禁止 | 仅限 trivial |

### 1.3 最高 ROI 与最高风险场景

**最高 ROI**：1. 依赖更新与 CI 自修复 2. 夜间特性工厂 3. 机械性重构 4. 测试生成 5. 文档同步
**最高风险**：1. 安全相关代码 2. 数据库迁移 3. 架构级重构 4. 性能关键路径

### 2.1 TDD-first 循环

### 2.2 Prompt Chaining

> 详细定义见 [01-core-specification.md](01-core-specification.md) 3.2。

编码阶段的链式执行：分析链 → 设计链 → 编码链 → 验证链。每步最大重试 3 次，超过转人工。

> 详细定义见 [01-core-specification.md](01-core-specification.md) 2.2。

**修复禁忌**：不得修改测试断言使其"通过"、不得删除失败测试、不得添加 `@skip`、不得吞异常。

- 成功模式 → `domain-knowledge/tech-stack/{stack}.md`

- 设计模式 → `domain-knowledge/project-specific/architecture-decisions.md`

## 第 3 章：定时任务与持续自主执行

### 3.1 三种定时任务

| 模式 | 执行环境 | 关机后运行 | 适用场景 |
|------|---------|-----------|---------|
| Cloud Scheduled | 云端 | 是 | 依赖更新、文档生成 |
| Desktop Scheduled | 本地 Desktop App | 否 | 构建、测试、Docker |
| CLI /loop | 会话级 | 否 | 轮询、持续监控 |

### 3.2 Kill Switch 层级

| 级别 | 触发条件 | 动作 |
|------|---------|------|
| L1 警告 | 运行>4h、成本>$10 | 通知但不中断 |
| L2 停止 | 运行>8h、成本>$30、修改 forbidden_dirs | 立即中断并通知 |
| L3 紧急 | 安全事件、数据泄露 | 中断 + 回滚 + 告警 |

### 3.3 异步 Decision Point

L3 夜间/周末自主开发时，人类审核者可能不在场，使用异步 Decision Point：

```
AI 到达 Decision Point → 生成 DP 摘要 → 通过通知渠道发送人工审核者
    ├── 阻塞模式：暂停等待确认（DP1 需求理解、DP2 架构方案）
    ├── 非阻塞模式：通知人工，继续执行，人工不认可则 revert（DP3 发布决策）
    └── 超时降级：2 小时未确认 → 自动降级为非阻塞模式 + 记录警告
```

**规则**：DP1/DP2 必须阻塞；通知必须包含完整上下文；人工确认必须具体，不得使用"OK"、"继续"等模糊回复。

### 3.4 夜间/周末开发前置检查清单

- [ ] DCP Go 已确认
- [ ] Spec 文件 ready（specs/ 中 status=ready）
- [ ] Kill Switch 参数已设置
- [ ] 通知 Channel 已测试
- [ ] Remote Control 已启用
- [ ] AC 覆盖率 100%（包覆盖率不得较基线下降 > 5%）

## 第 4 章：自修复 CI

### 4.1 什么能修

| 类别 | 自动修复 | 示例 |
|------|---------|------|
| 格式/命名/导入 | 是 | 缩进、未使用导入 |
| 类型错误 | 是（最小修改） | 缺少类型标注 |
| Lint 警告 | 是 | 代码风格 |
| 简单测试失败 | 是（最多 3 轮） | Mock 配置错误 |

### 4.2 什么必须转人工

安全漏洞、架构问题、数据迁移、性能问题、业务逻辑错误、Self-Correction > 3 轮、影响 > 50 个文件、涉及密钥配置。

### 4.3 Flaky Test 处理

识别条件：同一测试连续 3 次 CI：通过→失败→通过，或失败消息包含时序关键词。识别后记录到 `.flaky-tests.yml`，标记但不阻塞合并。

## 第 5 章：Supervisor-Worker 自动编排

### 5.1 角色定义

| 角色 | 模型 | 职责 |
|------|------|------|
| Supervisor | Opus | 需求分析、任务拆分、分发、冲突仲裁、质量判断 |
| Worker | Sonnet | 执行具体编码任务（TDD-first 循环） |

### 5.2 编排流程

1. **需求分析**：Supervisor 读取全部 ready Spec
2. **任务拆分**：构建依赖关系图，拆分为原子任务
3. **分发**：按依赖顺序分发给 Worker
4. **并行执行**：无依赖任务并行执行
5. **汇总**：收集所有 PR，运行集成测试
6. **质量判断**：全量测试 + 幻觉扫描 + 安全检查

## 第 6 章：Auto-Coding 度量

### 6.1 核心指标

| 指标 | 计算方式 | 目标 |
|------|---------|:----:|
| 自主成功率 | 一次通过审查的 PR 数 / 总 PR 数 | ≥ 70%（L3）/ ≥ 85%（L4） |
| 幻觉率 | 包含幻觉代码的 PR 数 / 总 PR 数 | < 5% |
| TDD 合规率 | 遵循 TDD 流程的 PR 数 / 总 PR 数 | ≥ 80% |
| 平均单次 PR 成本 | 总 API 成本 / PR 总数 | 持续下降趋势 |

### 6.2 预警规则

| 条件 | 动作 |
|------|------|
| 自主成功率 < 70% 连续 2 周 | 降级 L3→L2 |
| 幻觉率 > 10% | 暂停自主运行 |
| TDD 合规率 < 80% | 降级到 L1 |
| 检测到密钥 | 紧急停机 + 回滚 |
