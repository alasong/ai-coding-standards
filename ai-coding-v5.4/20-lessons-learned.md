# Lessons Learned Protocol

> 版本：v5.5 | 2026-04-24
> 定位：结构化沉淀失败经验和设计盲区，形成"教训链"（与"证据链"并列）
> 核心原则：**教训必须结构化、可检索、可注入，不能只留在对话记录里**

---

## 0. 为什么需要教训链

证据链（P11）记录"我们做了什么"——这是正向追溯。
教训链记录"我们做错了什么，下次不要这样做"——这是负向预防。

**没有教训链的后果（已发生）：**
- F015 泛化引擎只做"别名扩展"，未做"参数发现"。设计时没有参考生产中的真实意图分布。
- 竞品分析停留在"功能有无对比"，未拆解核心机制。深度问题直到用户直接指出才被发现。
- 同样的深度问题在不同 Feature 上重复出现，因为教训没有沉淀。

---

## 1. 教训的分类

| 级别 | 名称 | 定义 | 示例 |
|------|------|------|------|
| **L1 教训** | 设计盲区 | 设计阶段遗漏了关键维度 | F015 泛化只做了别名，没做实体识别 |
| **L2 教训** | 执行偏差 | 按规范执行但结果不对 | 竞品分析做了功能对照表，但没拆解机制 |
| **L3 教训** | 规范缺陷 | 规范本身有漏洞或不合理 | DCP 只检查"有没有做"，不检查"做得够不够深" |
| **L4 教训** | 系统性风险 | 体系级的结构性问题 | 生产数据不回灌到 Phase 0，导致设计脱离实际 |

---

## 2. 教训的结构

每条教训必须包含以下字段：

```yaml
id: LL-{NNN}
feature: "F015"
phase: "P0"  # 问题首次暴露的 Phase
level: L1    # L1-L4
title: "简短描述"
context: "当时在做什么"
what_happened: "发生了什么"
why_it_happened: "根因分析（至少追问 2 层 why）"
impact: "影响范围和严重程度"
prevention: "下次如何预防（具体动作）"
injected_into: "注入到了哪个规范/流程"
discovered_at: "2026-04-19"
discovered_by: "user review"
status: open  # open → injected → verified
verified_at: ""
```

## 3. 教训的生命周期

```
发现教训 → 记录到 lessons/ → 注入到规范/流程 → 后续 Feature 验证
                                                    ↓
                                          如果验证通过 → 标记 verified
                                          如果验证失败 → 回到注入步骤
```

### 3.1 发现时机

| 时机 | 触发者 | 动作 |
|------|--------|------|
| 用户直接指出 | 人类 | 立即记录，不辩解 |
| 深度评分发现 | P6 Agent | 标记为 `[DEPTH-GAP]` 的维度自动成为候选教训 |
| 对抗审查发现 | P3 Agent | 脆弱点清单中 ≥ HIGH 级别的自动成为候选教训 |
| 生产问题 | 运维/监控 | 从告警/事故中提取 |

### 3.2 注入规则

教训记录后必须在 **48 小时内** 注入到规范或流程中：

| 教训级别 | 注入目标 | 方式 |
|---------|---------|------|
| L1 设计盲区 | 对应 Phase 的深度评分维度 | 新增评分维度或扩大现有维度 |
| L2 执行偏差 | 对应 Phase 的方法论文档 | 补充"不要这样做"的反例 |
| L3 规范缺陷 | 相关规范文件 | 修订条款或新增约束 |
| L4 系统性风险 | 架构决策记录 + Phase 0 市场洞察 | 更新风险矩阵 |

### 3.3 验证规则

教训注入后，下一个 Feature 的 P6 深度评分必须检查：
1. 该教训是否已被正确注入
2. 注入后的机制是否有效拦截了同类问题
3. 如果拦截有效 → 标记 `verified`
4. 如果拦截无效 → 退回重新注入

## 4. 教训的检索与使用

### 4.1 检索方式

AI 在开始任何 Feature 开发前，必须执行：

```
1. 读取 lessons/lessons-registry.yaml
2. 筛选 status = open 或 verified 的教训
3. 筛选与本 Feature 相关的教训（相同 Phase 或相同模块）
4. 在 Spec 的 "已知风险" 章节引用相关教训 ID
```

### 4.2 强制引用

Spec 的 YAML frontmatter 新增字段：

```yaml
lessons_referenced:
  - LL-001: "泛化引擎不能只做别名扩展"
  - LL-002: "竞品分析必须拆解机制"
```

**验证：** P6 深度评分时检查是否引用了相关教训。如果有 open 状态的教训与本 Feature 相关但未被引用，深度评分标记 `[LESSONS-MISSING]`。

---

## 5. 教训与深度评分的联动

深度评分的 `self_check` 新增检查项：

| 检查项 | 检测逻辑 | 动作 |
|--------|---------|------|
| 教训引用完整性 | 相关 open 教训是否被引用 | 缺失 → `[LESSONS-MISSING]` |
| 教训拦截有效性 | 已 verified 教训是否真正拦截了同类问题 | 无效 → `[LESSONS-LEAK]` |
| 教训密度 | 当前 Feature 的教训数量 | > 3 条 → 标记该 Feature 为高风险 |

## 8. 教训的清理

教训不是永久的。满足以下条件可标记 `archived`：

| 条件 | 说明 |
|------|------|
| 连续 10 个 Feature 未触发 | 该教训预防的问题已彻底解决 |
| 规范已被根本性重构 | 原教训不再适用 |
| 教训本身被证明是错误的 | 复盘后发现不是真问题 |

archived 教训保留在 registry 中但标记为 `status: archived`，不再强制引用。

## 9. 教训密度指标

每月统计：

| 指标 | 计算 | 目标 |
|------|------|------|
| 新教训/Feature | 本月新发现教训数 / Feature 数 | 趋势下降（说明体系在成熟） |
| 教训注入率 | 已注入教训数 / 总教训数 | ≥ 90% |
| 教训验证率 | 已验证教训数 / 已注入教训数 | ≥ 80% |
| 教训泄漏率 | 已验证但同类问题再次发生的比例 | < 10% |
