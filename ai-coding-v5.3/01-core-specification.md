## 第 1 章：核心原则

### 1.1 核心原则清单（P1-P11，不可违反）

| # | 原则 | 说明 | Auto-Coding 含义 |
|---|------|------|-----------------|
| **P1** | **商业驱动** | 所有开发活动必须有明确的商业目标支撑 | 无商业目标的 AI 编码是纯浪费 |
| **P2** | **DCP 门禁** | 每个阶段必须通过决策门才能进入下一阶段 | L3/L4 可异步执行，但不得跳过 |
| **P3** | **TDD 先行** | AI 生成实现前必须先有测试，且测试必须先失败（Red） | L3/L4 下 AI 自动执行 Red→Green→Refactor，CI 必须记录 Red 状态 |
| **P4** | **人工审查** | 所有 AI 生成的代码必须经过人工 Code Review 才能合并 | L4 仅 trivial fix 可自动合并 |
| **P5** | **密钥不入代码** | 任何密钥、密码、token 不得出现在代码或配置文件中 | 必须配置 pre-commit hook + SAST 双重拦截 |
| **P6** | **单一信息源** | 每个事实只在一个地方定义，其他地方引用 | 过时文档=AI 按旧信息编码 |
| **P7** | **Spec 驱动** | AI 生成代码必须有明确的 Spec 文件作为输入 | L3/L4 下 Agent 直接从 `specs/` 读取任务队列 |
| **P8** | **最小批量** | AI 一次只生成一个函数或小模块的代码 | >50 行函数或 >200 行文件必须拆分 |
| **P9** | **Prompt 版本化** | 用于生成代码的 Prompt 必须版本化并可追溯 | L3/L4 下自动持久化到 `prompts/` |
| **P10** | **数据分级** | 发送给 AI 的数据必须经过分类，敏感数据禁止发送 | 必须实现 pre-send 扫描自动拦截 |
| **P11** | **证据链** | 每个 AI 声明必须有可验证的证据支撑，机器可验证、跨引用≥2 条、证据链不断裂 | L2+ 自动收集到 `.gate/` 目录 |

### 1.2 工程实践原则（P12-P23）

| # | 原则 | 说明 |
|---|------|------|
| **P12** | **环境一致性** | 开发环境必须通过声明式配置管理，不靠运行时重试 |
| **P13** | **错误不吞** | 所有错误必须处理或返回，禁止 `//nolint:errcheck` |
| **P14** | **租户隔离** | 禁止硬编码租户 ID，从请求上下文提取 |
| **P15** | **并发安全** | 全局可变状态必须使用锁或原子操作保护 |
| **P16** | **资源清理** | 所有 map/channel/goroutine 必须有清理机制 |
| **P17** | **输入校验** | 所有外部输入视为不可信，必须验证 |
| **P18** | **JSON 安全** | SSE/HTTP 响应数据必须用 `json.Marshal`，禁止拼接 |
| **P19** | **认证门禁** | 所有写操作端点必须有认证和角色校验 |
| **P20** | **速率保护** | 所有昂贵端点必须有速率限制 |
| **P21** | **数据一致性** | DB 写入失败必须返回错误，不得静默降级 |
| **P22** | **IP 不暴露** | 生产 IP/域名不得出现在代码中 |
| **P23** | **需求→Spec 链** | AI 生成代码前必须经过需求分析→架构适配→方案设计→Spec 生成四阶段链 |

### 1.3 P23：方案设计驱动详解

#### 1.3.1 Requirement→Solution→Spec 链

```
需求输入 → [需求分析] → [架构适配] → [方案设计] → [Spec 生成] → 编码执行
            DP0            DP0.5         DP0.7          DP1
```

| 阶段 | 输入 | 输出 | 验证 |
|------|------|------|------|
| **需求分析** | 用户原始需求 + 业务背景 | 结构化需求文档 | DP0：人工确认 |
| **架构适配** | 结构化需求 + 现有架构文档 | 架构适配分析 | DP0.5：人工确认 |
| **方案设计** | 架构适配分析 + 设计模板 | 方案设计文档 | DP0.7 + Solution Quality Gate |
| **Spec 生成** | 方案设计文档 + Spec 模板 | Spec 文件 | DP1：人工确认 |

#### 1.3.2 Context Loading Gate

进入需求分析前必须加载领域知识：

| # | 检查项 | 知识来源 |
|---|--------|---------|
| 1 | 行业领域知识 | `domain-knowledge/industry/{domain}.md` |
| 2 | 技术栈知识 | `domain-knowledge/tech-stack/{stack}.md` |
| 3 | 项目特定知识 | `domain-knowledge/project-specific/` |
| 4 | 现有架构理解 | `docs/architecture/` |
| 5 | 领域模型 | `docs/domain-model/` |

#### 1.3.3 Solution Quality Gate

方案设计完成、Spec 生成前必须通过 8 项检查：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | **需求覆盖** | 覆盖所有已确认的验收标准 |
| 2 | **架构一致性** | 与现有架构文档无冲突 |
| 3 | **接口完整性** | 所有新增/变更接口有明确输入输出定义 |
| 4 | **数据模型正确** | 有明确的迁移方案 |
| 5 | **异常处理** | 每个关键路径有异常处理策略 |
| 6 | **可测试性** | 每个验收标准有对应测试策略 |
| 7 | **依赖明确** | 新增依赖有版本约束和替换方案 |
| 8 | **风险评估** | 识别 Top 3 技术风险及缓解措施 |

#### 1.3.4 Skill Generalization

需求→Spec 链完成后触发：

1. **模式识别**：提取本次使用的设计/架构模式
2. **跨项目复用**：评估模式是否适用于其他项目
3. **知识沉淀**：将有效模式写入 `domain-knowledge/`
4. **反模式记录**：将失败方案写入 `historical-lessons.md`

### 1.4 自治等级模型（L1-L4）

| 等级 | 名称 | 人类干预点 | 适用场景 |
|------|------|-----------|---------|
| **L1** | 辅助编码 | 每一步 | 新手团队、安全敏感项目 |
| **L2** | 半自主编码 | 每个 PR 合并前 | **日常开发（推荐默认）** |
| **L3** | 受限自主编码 | PR 合并前 + DCP 门禁 | 夜间/周末开发、成熟团队 |
| **L4** | 完全自主编码 | DCP 门禁 + 定期审计 | 高成熟度团队、低风险变更 |

**升级路径**：L1→L2（≥20 PR 无事故）→L3（1 个月，≥70% 自主成功率）→L4（3 个月，≥85% 自主成功率）

**降级条件**（满足任一即降级）：

| 触发条件 | 降级目标 |
|---------|---------|
| 生产安全事故（AI 代码导致） | 降到 L2 |
| 幻觉代码合并到 main | 降到 L2 |
| 审计通过率连续 2 周 < 95%（L4） | 降到 L3 |
| 自主成功率连续 2 周 < 70%（L3） | 降到 L2 |
| TDD 执行率 < 80% | 降到 L1 |
| 密钥泄露到代码仓库 | 降到 L1 |

### 1.5 IPD 流程映射

| IPD Phase | 名称 | 自治等级影响 |
|-----------|------|-------------|
| Phase 0-2 | 市场洞察→详细规划 | 人类 + AI 协同，人类决策 |
| **Phase 3** | **AI 辅助开发** | **直接影响 — 决定 AI 自主程度** |
| Phase 4-5 | 验证发布→生命周期 | L4 下可部分自动化 |

## 第 2 章：TDD 与质量保障

### 2.1 Red → Green → Refactor 流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Red 阶段    │────▶│ Green 阶段   │────▶│ Refactor 阶段│
│  写测试      │     │ 写实现       │     │ 重构代码     │
│  运行 → 失败 │     │ 运行 → 通过  │     │ 运行 → 通过  │
└─────────────┘     └─────────────┘     └─────────────┘
```

| 阶段 | 要求 |
|------|------|
| **Red** | 测试文件必须先提交且运行失败，CI 记录 Red 状态 |
| **Green** | 实现提交后所有测试通过，CI 验证 Red→Green 转换 |
| **Refactor** | 重构后测试仍全部通过，CI 验证无回归 |

### 2.2 Self-Correction Loop（最多 3 轮）

```
AI 生成代码 → 自动化检查 → 通过→Code Review
                         → 失败→Round 1→通过→Code Review
                                    →失败→Round 2→通过→Code Review
                                               →失败→Round 3→通过→标记"3 轮后通过"
                                                          →失败→停止，通知人工
```

**适用场景**：Lint 错误（80%）、类型错误（85%）、测试失败（70%）、API 幻觉（60%）
**不适合**：架构不合理（20% 成功率）

### 2.3 全量验证 Gate

```
1. 编译/类型检查
2. 全量测试
3. lint
4. 失败 → Self-Correction Loop（最多 3 轮）
5. 3 轮后仍未通过 → 停止，向人类报告
```

**覆盖率规则**：全量运行，每包≥80%，证据文件写入 `.gate/coverage.json`
**AC 覆盖规则**：每个 AC 有≥1 测试函数 + ≥2 证据项，AC 覆盖率<100%=未完成

### 2.4 AI-First-Then-Human-Refine 模式

TDD 先行的补充，仅适用于：快速原型验证、探索性编码、UI/UX 原型。规则：必须在 PR 中说明原因，合并前仍须补齐测试，同一模块连续使用超过 3 次需启动根因分析。

## 第 3 章：AI 幻觉检测与防护

### 3.1 核心认识

AI 幻觉是 LLM 的本质特征，**每次生成都可能发生**。Gate 的存在不是为了检测是否发生，而是确保即使发生也不会进入生产。

### 3.2 防护策略

| 策略 | 说明 |
|------|------|
| **Example-Driven Prompting** | 用 `.examples/` 中的代码示例替代规则描述，AI 对示例的理解是规则的 5 倍 |
| **Prompt Chaining** | 将复杂任务分解为顺序链：分析→设计→编码→验证，每步验证输出质量 |
| **Progressive Disclosure** | 按需加载上下文，P0（Spec）每次加载，P1（相关源文件）首次提到时加载，P2（项目历史）手动加载 |
| **AI Reviewer + Human Reviewer** | 两层审查，AI Reviewer 检测幻觉，Human Reviewer 专注业务逻辑 |

> **前置链**：Prompt Chaining 用于编码执行阶段，必须先完成 P23 的 Requirement→Solution→Spec 链（编码前）。

### 3.3 检测方法

| 方法 | 检测的幻觉类型 |
|------|---------------|
| 编译/类型检查 | API 幻觉、类型不匹配 |
| 符号解析 | 未定义的符号引用 |
| 依赖验证 | 依赖幻觉、版本幻觉、Typosquatting |
| SAST 扫描 | 安全幻觉、漏洞 |
| AC 映射验证 | 覆盖率幻觉、完成幻觉 |

### 3.4 指标

| 指标 | 目标 | 告警阈值 |
|------|:----:|---------|
| 幻觉发生率 | < 5% | > 10% |
| 幻觉拦截率 | 100% | < 90% |
| 幻觉逃逸率 | 0% | > 0% |

## 第 4 章：Spec 驱动开发

### 4.1 Feature Spec 规范

每个 Feature 必须有且仅有一个 Spec 文件，存放在 `specs/` 目录：

```yaml
---
type: feature-spec
id: F001
name: "用户注册"
version: 1.0
status: draft | validated | ready | in-progress | done | deprecated
priority: must | should | could | wont
validated_by: "@name"
required_context:
  - docs/architecture.md#L50-L80
autonomy_level: L2
---
```

Spec 必须包含：用户故事、验收标准（Gherkin 格式）、边界条件、数据模型、API 设计、非功能需求。

### 4.2 Spec Validation Gate

| 验证项 | 检查要点 |
|--------|---------|
| 与 PRD 一致性 | 每个 PRD 需求至少有一个 AC 对应 |
| 与架构一致性 | 不违反任何 ADR |
| 验收标准可测试 | Gherkin 格式可执行 |
| 边界条件完整 | 至少覆盖正常路径 + 3 种异常路径 |
| 非功能需求明确 | 数值化，不可用模糊描述 |

### 4.3 Spec 状态机

```
draft →(人工审核)→ validated →(放入队列)→ ready →(AI 开发)→ in-progress →(Gate+Review)→ done
                                                                 any →(取消)→ deprecated
```

## 第 5 章：Prompt 管理

### 5.1 Prompt 版本化（P9）

每个用于生成代码的 Prompt 必须存储在 `prompts/` 目录，包含 YAML frontmatter（id、version、model、temperature、feature、status）。同一 Prompt 每次修改递增版本号。Prompt 一次通过率 < 50% = 需要重写。

### 5.2 Prompt 安全

- Prompt 文件必须来自本地仓库，禁止从 URL 或外部服务加载
- Spec 中的用户输入不得直接拼接到 Prompt 中
- 每季度审查 Prompt 仓库，删除过时 Prompt

## 第 6 章：人与 AI 的分工

### 6.1 基础分工

| 活动 | 人类角色 | AI 角色 |
|------|---------|---------|
| 战略/产品/架构决策 | **决策者** | 信息整理、方案对比 |
| Spec 编写 | 审核者 | 草稿生成（L2+） |
| 测试编写/编码实现 | 审核者 | **执行者** |
| 安全审查 | **决策者** | 辅助扫描 |
| 部署运维 | 监督者（L1-L2）/ 审计者（L4） | **执行者** |

### 6.2 Decision Point

| # | 决策点 | 时机 | 拦截的错误类型 |
|---|--------|------|---------------|
| **DP0** | 需求确认 | 需求分析完成后 | 需求理解偏差 |
| **DP0.5** | 架构方向 | 架构适配完成后 | 架构不兼容 |
| **DP0.7** | 方案可行性 | 方案设计完成后 | 方案不可行 |
| **DP1** | 需求理解 | AI 分析完代码结构后 | AI 误解需求 |
| **DP2** | 架构方案 | AI 给出技术方案后 | 过度工程 |
| **DP3** | 发布决策 | 全部完成后 | 功能不完整 |
| **DP4** | 紧急变更 | Hotfix 时 | 修复引入新问题 |

**L3 异步模式**：DP1/DP2 阻塞，DP3 非阻塞，超时 2 小时自动降级为非阻塞。

### 6.3 DCP 与 Decision Point 的关系

| | DCP | Decision Point |
|---|---|---|
| 粒度 | 阶段级 | 任务级 |
| 频率 | 每个 Phase 一次 | 每个任务多次 |
| 关系 | 宏观保护 | 微观保护 |

两者互补，不可替代。
