## 第 1 章：45 种幻觉类型（8 大类）

### 1.1 存在性幻觉（E 系列）

| 编号 | 类型 | 说明 |
|------|------|------|
| E01 | API 幻觉 | 调用不存在的函数/方法/API |
| E02 | 依赖幻觉 | 引用不存在的依赖包或版本号 |
| E03 | 文件幻觉 | 引用不存在的文件路径 |
| E04 | 变量幻觉 | 使用未定义的变量或字段 |
| E05 | 配置幻觉 | 引用不存在的配置项或环境变量 |
| E06 | 路径幻觉 | 使用不存在或不正确的文件路径 |

### 1.2 执行性幻觉（X 系列）

| 编号 | 类型 | 说明 |
|------|------|------|
| X01 | 操作幻觉 | 执行了不存在的操作 |
| X02 | 部分执行 | 声称完成但只完成部分 |
| X03 | 跳过步骤 | 跳过了必要的执行步骤 |
| X04 | 重复执行 | 重复执行了不必要的操作 |
| X05 | 回滚幻觉 | 声称回滚但实际未回滚 |

### 1.3 验证性幻觉（V 系列）

| 编号 | 类型 | 说明 |
|------|------|------|
| V01 | 测试幻觉 | 生成能通过但需求未满足的测试 |
| V02 | 选择性验证 | 只验证了成功的部分 |
| V03 | 编译幻觉 | 声称编译通过但实际失败 |
| V04 | **覆盖率幻觉** | 把"包通过"等同于"需求覆盖" |
| V05 | Lint 幻觉 | 声称 lint 通过但实际有警告 |
| V06 | 安全验证幻觉 | 声称安全扫描通过但实际有漏洞 |

### 1.4 逻辑性幻觉（L 系列）

## 第 2 章：证据链（Chain of Evidence）

### 2.1 5 层证据架构

```
意图证据 → 计划证据 → 行动证据 → 验证证据 → 审查证据
  (Spec)     (Design)    (Code)     (.gate/)    (Review)
```

### 2.2 证据要求

| 要求 | 说明 |
|------|------|
| 机器可验证 | 证据必须来自自动化工具输出 |
| 跨引用≥2 条 | 每个声明至少 2 条独立证据 |
| 证据链不断裂 | 5 层证据必须完整 |
| 反面证据 | 失败也是证据，必须记录 |

### 2.3 AC 映射验证

每个验收标准（AC）必须有：
- ≥1 对应的测试函数
- ≥2 项证据（代码存在 + `.gate/` 中的运行输出）

**完成判定**：AC 覆盖率 < 100% = 功能未完成，不得标记 Feature 为 `done`。

## 第 3 章：检测与防护

### 3.1 检测链

```
编译验证 → 符号解析 → 依赖验证 → AI Reviewer → Human Reviewer → AC 映射验证
```

### 3.2 防护策略

| 策略 | 效果 |
|------|------|
| Example-Driven Prompting | 用示例替代规则，幻觉率降低 5 倍 |
| Prompt Chaining | 分步验证，质量提升 1.3-1.8 倍 |
| Progressive Disclosure | 按需加载上下文，保持高注意力 |
| 两层审查 | AI Reviewer + Human Reviewer，比单层好 40% |

### 3.3 应急响应

```
检测 → 立即阻止合并 → 分析幻觉类型 → 修复代码 → 强化 Gate → 记录到 known-issues.yaml
```

| 场景 | 应急动作 |
|------|---------|
| 幻觉代码合并到 main | 立即 revert → 根因分析 → 强化审查 |
| 幻觉导致安全漏洞 | 立即回滚 → 安全事件响应 → 全面审计 |
| 幻觉引入恶意依赖 | 立即删除依赖 → 依赖审计 → 供应链安全审查 |
