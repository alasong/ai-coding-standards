## 第 1 章：核心原则（P1-P11，不可违反）与工程实践原则（P12-P22）

### 1.1 什么是 AI Coding

**AI Coding** 是指使用大型语言模型（LLM）辅助完成软件开发生命周期中的任何活动。

AI Coding 与传统开发的本质区别：

| 维度 | 传统开发 | AI Coding |
|------|---------|----------|
| 执行者 | 人 | AI（人审核） |
| 工具性质 | 确定性（编译器、IDE） | 概率性（同一 prompt 可能产生不同输出） |
| 质量假设 | 人写的代码质量相对稳定 | 同一段 prompt 可能产生高质量代码，也可能产生幻觉代码 |
| 流程设计 | 假设确定性 | **必须处理不确定性** |

**核心判断**：AI 不是替代人的判断，而是放大人的执行力。好的判断 + AI = 10 倍产出；差的判断 + AI = 10 倍的错误。

### 1.2 核心原则清单

以下 11 条原则是 v5.1 的**绝对底线**，任何自治等级下都不可违反。每条原则包含"Auto-Coding 含义"，说明在自动编码场景下的具体约束。

| # | 原则 | 说明 | 违反后果 | Auto-Coding 含义 |
|---|------|------|---------|-----------------|
| **P1** | **商业驱动** | 所有开发活动必须有明确的商业目标支撑 | 浪费资源，产出无价值 | AI 可以 24/7 编码，但必须有人定义"值得做什么"。无商业目标的 AI 编码是纯浪费 |
| **P2** | **DCP 门禁** | 每个阶段必须通过决策门才能进入下一阶段 | 在错误方向上投入 | Auto-Coding 模式下，DCP 可以异步执行（见第 7 章），但不得跳过。夜间/周末开发尤其需要 DCP 前置 |
| **P3** | **TDD 先行** | AI 生成实现前必须先有测试，且测试必须先失败（Red） | AI 代码无质量保障 | L3/L4 自治等级下，AI 自动执行 Red→Green→Refactor 全流程，CI 必须记录 Red 状态，禁止 TDD 造假 |
| **P4** | **人工审查** | 所有 AI 生成的代码必须经过人工 Code Review 才能合并 | AI 幻觉代码进入生产 | L1-L3：每个 PR 合并前必须有人工审查；L4：允许定期审计替代逐 PR 审查，但 trivial fix 之外的变更仍需人工审查 |
| **P5** | **密钥不入代码** | 任何密钥、密码、token 不得出现在代码或配置文件中 | 安全泄露 | AI 倾向于从上下文复制密钥到代码中。必须配置 pre-commit hook + SAST 双重拦截，不依赖 AI 自觉 |
| **P6** | **单一信息源** | 每个事实只在一个地方定义，其他地方引用 | 信息不一致，决策错误 | Auto-Coding 下，AI 读取的上下文文件（AGENTS.md、architecture.md）是事实的唯一来源。过时文档=AI 按旧信息编码 |
| **P7** | **Spec 驱动** | AI 生成代码必须有明确的 Spec 文件作为输入 | AI 理解偏差，方向性错误 | L3/L4 下，Agent 直接从 `specs/` 目录读取任务队列自动执行，Spec 质量直接决定 Auto-Coding 质量 |
| **P8** | **最小批量** | AI 一次只生成一个函数或小模块的代码 | 大规模错误代码，难以定位 | AI 的注意力窗口有限。超过 50 行的函数或超过 200 行的文件必须拆分。批量越大，幻觉概率越高 |
| **P9** | **Prompt 版本化** | 用于生成代码的 Prompt 必须版本化并可追溯 | 无法复现、审计、调试 AI 输出 | L3/L4 下，Prompt 必须自动持久化到 `prompts/`，每个 PR 声明使用的 Prompt 版本，支持回滚和复现 |
| **P10** | **数据分级** | 发送给 AI 的数据必须经过分类，敏感数据禁止发送 | 数据泄露给 AI 供应商 | 必须实现 pre-send 扫描，自动拦截 Restricted 级别数据。AI 不知情=人不负责=泄露发生 |
| **P11** | **证据链** | 每个 AI 声明必须有可验证的证据支撑。证据必须机器可验证、跨引用至少 2 条、证据链不可断裂 | 无法证明功能真正实现 | L2+：AI 自动收集证据到 `.gate/` 目录。每个验收标准（AC）必须有 ≥1 测试函数和 ≥2 证据项。报告必须区分包通过率/AC 覆盖率/端点通过率 |

### 1.3 原则分级与自治等级关系

11 条核心原则在任何自治等级下都不可违反，但**执行方式**随等级变化：

- **L1（辅助编码）**：人工执行每一步检查和确认
- **L2（半自主编码）**：AI 执行生成和初步验证，人工审查每个 PR
- **L3（受限自主）**：AI 执行完整开发循环，人工在 PR 合并前审查 + DCP 门禁
- **L4（完全自主）**：AI 执行全部工作，定期审计 + 自动门禁，仅 trivial fix 可自动合并

详细约束矩阵见第 2 章。

### 1.4 自治等级与 IPD 流程的映射

v5.0 的自治等级模型与 v4 的 IPD 6 阶段流程协同工作。自治等级**仅影响 Phase 3（AI 辅助开发）阶段的执行方式**，不影响其他阶段的决策流程。

| IPD Phase | 名称 | 自治等级影响 |
|-----------|------|-------------|
| Phase 0 | 市场洞察 | 人类 + AI — AI 辅助数据收集与趋势分析，人类决策 |
| Phase 1 | 概念定义 | 人类 + AI — AI 辅助竞品分析与需求建模，人类决策 |
| Phase 2 | 详细规划 | 人类 + AI — AI 辅助架构分析与方案设计，人类决策 |
| **Phase 3** | **AI 辅助开发** | **直接影响 — 决定 AI 自主程度** |
| Phase 4 | 验证发布 | L4 下可部分自动化（ADCP 检查清单自动化） |
| Phase 5 | 生命周期 | L4 下可自动收集指标、生成报告 |

每条原则在不同自治等级下的违规后果：

### 1.6 工程实践原则（P12-P22）

P1-P11 是不可违反的核心原则。P12-P22 是工程实践原则，通过**多层执行机制**（Layered Enforcement）保障——不是靠 AI 自觉，而是靠工具链自动拦截。

| # | 原则 | 说明 | 违反后果 | Auto-Coding 含义 |
|---|------|------|---------|-----------------|
| **P12** | **环境一致性** | 开发环境必须通过声明式配置管理，不同语言使用各自生态最佳实践，不靠运行时重试 | 开发者无法启动、CI 失败、环境不一致 | AI 编码假设环境已就绪，不在运行时处理环境问题。环境配置必须声明式、可复现、语言适配 |
| **P13** | **错误不吞** | 所有 `json.Marshal`/`json.Encode`/DB 写入/文件写入错误必须处理或返回 | 运行时静默失败，问题定位困难 | 禁止使用 `//nolint:errcheck`、`_ = json.Marshal(...)` 等吞错误模式 |
| **P14** | **租户隔离** | 禁止硬编码租户 ID，租户标识必须从请求上下文提取 | 数据泄漏、多租户污染 | 代码中不得出现 `"tn-default"` 等硬编码租户字符串 |
| **P15** | **并发安全** | 全局可变状态必须使用 `sync.RWMutex` 或 `atomic.Value` 保护 | 数据竞争、不可复现的偶发 Bug | 任何被多个 goroutine 读写的变量必须加锁 |
| **P16** | **资源清理** | 所有 map/channel/goroutine 必须有清理机制或大小限制 | 内存泄漏、goroutine 泄漏 | 全局缓存必须设 TTL/上限并启动后台清理 |
| **P17** | **输入校验** | 所有 HTTP 输入参数必须验证类型和范围 | 400 请求被接受导致异常行为 | 不得信任调用方输入，所有外部输入视为不可信 |
| **P18** | **JSON 安全** | SSE/HTTP 响应数据必须用 `json.Marshal`，禁止 `fmt.Sprintf` 拼接 | 无效 JSON 导致客户端解析失败 | SSE `data:` 后面的内容必须是合法 JSON |
| **P19** | **认证门禁** | 所有写操作端点必须有认证和角色校验 | 未授权数据修改 | GET 可跳过认证，POST/PUT/DELETE 必须有角色校验 |
| **P20** | **速率保护** | 所有昂贵端点（LLM 调用、DB 批量写入）必须有速率限制 | DoS 攻击、资源耗尽 | 无 rate limiter 的端点不得上线 |
| **P21** | **数据一致性** | DB 写入失败必须返回错误给调用方，不得静默降级 | 数据不一致，调用方误以为成功 | 不得用空返回值掩盖写失败 |
| **P22** | **IP 不暴露** | 生产 IP/域名不得出现在代码中，必须用环境变量 | 配置泄漏、环境切换困难 | 已知生产地址必须走 `.env` 或 `config.yaml` |
| **P23** | **需求→Spec 链** | AI 生成代码前必须经过需求分析→架构适配→方案设计→Spec 生成四阶段链，不得跳过方案设计直接编码 | 方向性错误、方案与架构不符、Spec 质量低下导致后续全链错误 | L3/L4 下 AI 自动执行全链，每个阶段产出物必须通过对应 Decision Point（DP0/DP0.5/DP0.7/DP1）才能进入下一阶段 |

### 1.6.2 工程实践 P23：方案设计驱动的 Auto-Coding

P23 的核心认识：**高质量的 Spec 来自高质量的方案设计，而高质量的方案设计来自对需求的深度理解和对现有架构的适配分析。** 跳过方案设计直接编码，是 AI 编码中最常见的方向性错误。

#### 2.24.1 Requirement→Solution→Spec 链

```
需求输入 → [需求分析] → [架构适配] → [方案设计] → [Spec 生成] → 编码执行
            DP0            DP0.5         DP0.7          DP1
```

每个阶段的输入和输出：

| 阶段 | 输入 | 输出 | 验证 |
|------|------|------|------|
| **需求分析** | 用户原始需求 + 业务背景 | 结构化需求文档（用户故事 + 验收标准） | DP0：人工确认需求理解正确 |
| **架构适配** | 结构化需求 + 现有架构文档 | 架构适配分析（接口变更 + 数据模型变更 + 依赖变更） | DP0.5：人工确认架构方向正确 |
| **方案设计** | 架构适配分析 + 设计模板 | 方案设计文档（模块划分 + 接口定义 + 数据流 + 异常处理） | DP0.7 + Solution Quality Gate |
| **Spec 生成** | 方案设计文档 + Spec 模板 | Spec 文件（验收标准 + 实现约束 + 测试要求） | DP1：人工确认 Spec 完整 |

#### 2.24.2 链配置（`.aicoding.yaml`）

```yaml
requirement_to_spec_chain:
  phases:
    - name: requirement_analysis
      output: docs/requirements/{feature-id}-requirements.md
      decision_point: DP0
    - name: architecture_adaptation
      output: docs/solutions/{feature-id}-architecture.md
      decision_point: DP0.5
    - name: solution_design
      output: docs/solutions/{feature-id}-design.md
      decision_point: DP0.7
      quality_gate: solution-quality-gate
    - name: spec_generation
      output: specs/{feature-id}-spec.md
      decision_point: DP1
  max_retries_per_phase: 3
  fail_action: escalate_to_human
```

#### 2.24.3 Context Loading Gate（上下文加载门禁）

在进入需求分析阶段前，必须先加载领域知识，确保 AI 理解项目上下文。

**Context Loading Gate 检查项**：

| # | 检查项 | 说明 | 知识来源 |
|---|--------|------|---------|
| 1 | 行业领域知识 | 所在行业的基本规则、术语、合规要求 | `domain-knowledge/industry/{domain}.md` |
| 2 | 技术栈知识 | 项目使用的语言、框架、核心库的最佳实践和常见陷阱 | `domain-knowledge/tech-stack/{stack}.md` |
| 3 | 项目特定知识 | 项目特有的架构约定、命名规范、历史决策记录 | `domain-knowledge/project-specific/` |
| 4 | 现有架构理解 | 当前系统的模块划分、接口定义、数据流图 | `docs/architecture/` |
| 5 | 领域模型 | 核心业务实体及其关系 | `docs/domain-model/` |

**知识目录结构**：

```
domain-knowledge/
├── industry/
│   ├── fintech.md          # 金融科技领域知识
│   ├── healthcare.md       # 医疗健康领域知识
│   └── e-commerce.md       # 电商领域知识
├── tech-stack/
│   ├── go-gin.md           # Go + Gin 框架知识
│   ├── python-fastapi.md   # Python + FastAPI 知识
│   └── react-typescript.md # React + TypeScript 知识
└── project-specific/
    ├── architecture-decisions.md  # 架构决策记录
    ├── naming-conventions.md      # 命名规范
    └── historical-lessons.md      # 历史教训记录
```

#### 2.24.4 Solution Quality Gate（方案质量门禁）

在方案设计完成、Spec 生成前，必须通过 8 项质量检查。

| # | 检查项 | 通过标准 | 失败后果 |
|---|--------|---------|---------|
| 1 | **需求覆盖** | 方案设计覆盖所有已确认的验收标准 | 转回需求分析阶段 |
| 2 | **架构一致性** | 与现有架构文档无冲突，或冲突已明确标注并获批准 | 转回架构适配阶段 |
| 3 | **接口完整性** | 所有新增/变更的接口有明确的输入输出定义 | 补充接口定义 |
| 4 | **数据模型正确** | 数据模型变更有明确的迁移方案 | 补充迁移方案 |
| 5 | **异常处理** | 每个关键路径有明确的异常处理策略 | 补充异常处理 |
| 6 | **可测试性** | 每个验收标准有对应的测试策略 | 补充测试策略 |
| 7 | **依赖明确** | 新增依赖有版本约束和替换方案 | 明确依赖约束 |
| 8 | **风险评估** | 识别了 Top 3 技术风险及缓解措施 | 补充风险评估 |

#### 2.24.5 Skill Generalization（技能泛化）

在完成需求→Spec 链后，AI 应执行技能泛化分析，提取可复用的模式：

1. **模式识别**：本次方案设计中使用的设计模式、架构模式
2. **跨项目复用**：该模式是否适用于其他项目或模块
3. **知识沉淀**：将验证有效的模式写入 `domain-knowledge/` 目录
4. **反模式记录**：将本次尝试但失败的方案写入 `historical-lessons.md`

**Skill Generalization 触发时机**：每个 Feature 的需求→Spec 链完成后、进入编码阶段前。

### 1.6.1 P12 详解：语言适配的开发环境策略

P12 要求**声明式环境管理**，但不同语言的工具链特征不同，必须使用各自生态的最佳实践：

| 语言 | 工具链特征 | 推荐方案 | 配置文件 | AI 安装权限 |
|------|-----------|---------|---------|:----------:|
| **Go** | 编译器单二进制、依赖编译进二进制 | mise / 原生安装 | `.mise.toml` | ✅ 无需 sudo |
| **Node.js** | 运行时多版本、npm 依赖树大 | nvm / volta + Node 容器 | `.nvmrc` / `.voltrc` | ✅ 无需 sudo |
| **Python** | 包依赖图复杂、C 扩展多 | uv / venv + pip | `pyproject.toml` | ✅ venv 隔离 |
| **Rust** | 官方工具链完善 | rustup + cargo | `rust-toolchain.toml` | ✅ 无需 sudo |
| **多语言混合** | 跨语言依赖链 | Dev Container / Nix | `devcontainer.json` / `shell.nix` | ⚠️ 容器内 root |

**选择依据**：
1. 单语言项目 → 优先使用该语言的官方版本管理器
2. AI 能否独立安装 → 方案必须支持无 sudo 执行
3. 版本隔离需求 → 工具安装不应影响系统全局
4. 多语言项目 → Dev Container 统一环境

**Go 项目标准（本规范适用）**：
- 声明文件：`.mise.toml`（项目根目录）
- 安装命令：`mise install`（无需 sudo，装到 `~/.local/share/mise/installs/`）
- 引导脚本：`make setup`（安装 mise 自身 + 调用 `mise install`）
- CI 验证：`make check-env`（检测 go、golangci-lint 是否可用）
- Docker 角色：部署验证工具，不是日常开发环境

### 1.6.3 Docker 容器调试策略

Docker 容器作为**部署验证环境**，AI 必须能够独立完成诊断和修复。核心原则：

**诊断先于修改**：Docker build/runtime 失败时，必须先通过日志和状态定位根因，不得猜测修改 Dockerfile。

**Go 项目 Docker 调试标准**：

| 诊断目标 | 命令 | 说明 |
|---------|------|------|
| 查看详细构建 | `make docker-build-verbose` | `--progress=plain --no-cache` 显示每步完整输出 |
| 查看日志 | `make docker-logs` | 最近 100 行日志 |
| 进入容器 | `make docker-sh` | 在运行中的容器内执行 shell |
| 检查状态 | `docker compose ps` | 容器状态、健康检查、端口映射 |
| 验证配置 | `docker compose config` | 检查 docker-compose.yml 是否有效 |

**AI 调试流程（Build 失败时）**：
1. `make docker-build-verbose` → 查看哪一步 Step 失败
2. 如果 Step 是 `COPY` → 检查源文件路径是否存在，是否被 `.dockerignore` 排除
3. 如果 Step 是 `RUN go mod download` → 检查网络代理/GOPROXY，不要改 go.mod
4. 如果 Step 是 `RUN go build` → 先本地 `make build` 确认不是代码问题

**AI 调试流程（Runtime 失败时）**：
1. `make docker-logs` → 查看容器启动日志
2. `docker compose ps` → 确认容器状态（running / restarting / exited）
3. `make docker-sh` → 进入容器确认二进制文件在 `/app/skillmesh`
4. 检查 HTTP：`docker compose exec skillmesh wget -qO- http://localhost:8080/health`
5. 检查 DB：`docker compose exec postgres pg_isready -U skillmesh`

### 1.7 多层执行机制（Layered Enforcement）

P12-P22 不是靠 AI 自觉执行，而是通过以下四层机制自动拦截：

| 层级 | 机制 | 示例 | 拦截能力 |
|------|------|------|---------|
| **L1 AI 约束** | 规范声明，AI 生成时遵守 | AGENTS.md 中 P13-P22 声明 | 弱——AI 可能遗忘或疏忽 |
| **L2 Pre-commit** | Git hook 自动拦截 | 拦截 `//nolint:errcheck`、`"tn-default"` | 强——提交时自动阻断 |
| **L3 CI Gate** | `make check` 门禁 | `go test -race`、`check-env`、lint | 强——CI 自动阻断 |
| **L4 Runtime** | 代码本身防护 | `writeJSON` 返回 500、cleanup goroutine | 强——运行时兜底 |

**执行原则**：一条规则至少有两层保护（L1 + L2/L3/L4 之一）。只有 L1 的规则视为"未落地"。

## 第 2 章：自治等级模型（L1-L4，v5 新增）

### 2.1 为什么需要自治等级模型

v4.0 定义了"人与 AI 的分工"，但没有定义"AI 可以自主到什么程度"。在实际应用中，不同项目、不同阶段、不同团队成熟度下，AI 的自主程度应该不同。

**自治等级模型解决的核心问题**：
- 什么时候可以让 AI 独立完成一个 Feature？
- 什么时候必须人工介入每一个决策？
- 如何安全地从低等级升级到高等级？
- 什么情况下必须降级？

### 2.2 等级定义

v5.0 定义 4 个自治等级，**每个等级继承上一级的所有约束**：

| 等级 | 名称 | 人类干预点 | 适用场景 | 推荐度 |
|------|------|-----------|---------|--------|
| **L1** | 辅助编码 | 每一步（Spec 编写、测试编写、代码生成、Code Review、合并） | 新手团队、安全敏感项目、初期探索 | 过渡性 |
| **L2** | 半自主编码 | 每个 PR 合并前（AI 完成 Spec→测试→实现→自修，人工审查后合并） | **日常开发（推荐默认等级）** | 默认 |
| **L3** | 受限自主编码 | PR 合并前审查 + DCP 门禁（AI 可读取 specs/ 队列自主开发，夜间/周末持续工作） | 夜间/周末开发、自修复 CI、成熟团队 | 进阶 |
| **L4** | 完全自主编码 | DCP 门禁 + 定期审计（AI 独立完成全部开发循环，仅 trivial fix 可自动合并到 main） | 高成熟度团队、低风险变更、长期运维 | 高级 |

### 2.3 等级详细说明

#### L1：辅助编码

AI 作为"智能助手"，人在每一个步骤都参与决策：

- **Spec 编写**：AI 生成草稿，人工审核并修改
- **测试编写**：AI 基于 Spec 生成测试，人工审核断言正确性
- **代码生成**：AI 基于 Prompt 生成代码，人工逐行审查
- **自修循环**：AI 修复 lint/test 错误，人工确认修复结果
- **合并**：人工执行 git merge

**适用条件**：
- 团队刚开始使用 AI Coding，尚未积累信任
- 安全敏感项目（金融、医疗、政府）
- 法规要求人工逐行审查的场景

#### L2：半自主编码（推荐默认）

AI 作为"执行者"，人作为"审核者"：

- **Spec 编写**：AI 生成，人工审核（同 L1）
- **开发循环**：AI 独立完成 Red→Green→Refactor→Self-Correction（最多 3 轮）
- **Code Review**：两层审查（AI Reviewer + Human Reviewer）
- **合并**：人工执行，不得自动合并

**关键约束**：
- 每个 PR 必须有完整的人类审查签名
- AI Reviewer 检查 AI 特有问题（幻觉检测）
- Human Reviewer 专注于业务逻辑和架构约束
- 两层审查比单层审查好 40%（v4.0 数据）

**适用条件**：
- 团队积累了至少 20 个 PR 无安全事故
- TDD 执行率 > 80%
- Prompt 一次通过率 > 50%

#### L3：受限自主编码

AI 作为"夜班开发者"，人作为"早班审核者"：

- **任务获取**：AI 从 `specs/` 目录读取状态为 `ready` 的 Spec 自动开始
- **开发循环**：AI 独立完成全部开发（Spec 分析→测试→实现→自修→验证 Gate）
- **决策点**：DP1（需求理解）、DP2（架构方案）通过异步消息通知人工确认
- **PR 创建**：AI 自动创建 PR，填入完整的 Spec/Prompt/Model 追溯信息
- **合并**：人工审查 PR 后合并（可以批量审查多个 PR）
- **DCP 门禁**：夜间开发前必须完成 DCP 确认，DCP 可以异步执行

**关键约束**：
- 夜间/周末开发前必须通过 DCP Go 决策
- 每个 PR 必须经过人工审查后才能合并
- 幻觉检测 Gate 必须自动通过 CI 执行
- 发现安全问题立即暂停自主开发并降级到 L2

**适用条件**：
- L2 稳定运行至少 1 个月
- 自主成功率（PR 一次通过人工审查的比例）> 70%
- 幻觉发生率 < 5%
- 有完善的 CI Gate 和监控体系

#### L4：完全自主编码

AI 作为"独立开发者"，人作为"审计者"：

- **全部自主**：AI 独立完成 Spec 分析、测试、实现、自修、验证
- **自动合并**：仅限 trivial fix（lint 修复、格式修复、注释更新）可自动合并到 main
- **定期审计**：每周人工审计随机抽样的 PR（至少 10%）
- **DCP 门禁**：简化为自动化检查清单，重大变更仍需人工 DCP
- **降级触发**：审计通过率 < 95% 或出现安全事件立即降级到 L2

**关键约束**：
- 自动合并仅限以下类型：lint fix、format、typo fix in comments、dependency version patch update
- 任何涉及业务逻辑、架构变更、API 变更的代码不得自动合并
- 每周审计报告必须公开，团队可见
- 安全扫描 CRITICAL 级别问题必须立即阻断

**适用条件**：
- L3 稳定运行至少 3 个月
- 自主成功率 > 85%
- 审计通过率 > 95%
- 零安全事故记录
- 完善的自动化 Gate 体系和回滚机制

### 2.4 约束矩阵

每个核心原则在 L1-L4 下的具体执行方式：

| 约束 | L1 辅助编码 | L2 半自主编码 | L3 受限自主编码 | L4 完全自主编码 |
|------|-----------|-------------|---------------|---------------|
| **P1 商业驱动** | 人工定义每个商业目标 | 人工定义，AI 辅助拆解 | 人工定期定义，AI 按 Spec 队列执行 | 人工定期审查，AI 自主按优先级执行 |
| **P2 DCP 门禁** | 人工执行，同步确认 | 人工执行，同步确认 | 人工执行，可异步确认（夜间开发前） | 自动化检查清单 + 人工定期审计 |
| **P3 TDD 先行** | 人工确认 Red→Green | AI 执行，人工确认断言 | AI 自动执行，CI 记录 Red→Green | AI 自动执行，CI 强制验证 |
| **P4 人工审查** | 每次提交前人工审查 | 每个 PR 合并前人工审查 | 每个 PR 合并前人工审查（可批量） | 定期审计（每周，随机抽样 ≥10%） |
| **P5 密钥不入代码** | pre-commit hook 拦截 | pre-commit hook + SAST 双重拦截 | pre-commit hook + SAST + CI 自动扫描 | 同 L3 + 审计期专项密钥检测 |
| **P6 单一信息源** | 人工维护文档一致性 | 人工维护，AI 辅助检测过时文档 | AI 自动检测文档/代码漂移，人工确认 | 同 L3 + 自动漂移检测（>20% 触发回归） |
| **P7 Spec 驱动** | 人工编写 Spec | 人工审核 AI 生成的 Spec | AI 从 specs/ 读取，Spec 预先人工验证 | AI 从 specs/ 读取，Spec 定期人工抽检 |
| **P8 最小批量** | 人工控制每次生成的粒度 | AI 控制，人工审查粒度 | AI 控制，CI 验证函数/文件大小 | AI 控制，CI 强制验证 |
| **P9 Prompt 版本化** | 人工记录 Prompt 版本 | AI 自动记录到 PR 描述 | AI 自动持久化到 prompts/ 目录 | 同 L3 + 自动回归测试 |
| **P10 数据分级** | 人工判断数据级别 | pre-send 自动扫描 | pre-send 自动扫描 + 审计日志 | 同 L3 + 定期数据安全审计 |
| **P11 证据链** | 人工逐项核对 AC | AI 收集证据到 .gate/ | AI 自动收集 + AC 映射验证 | AI 自动收集 + 定期审计 |
| **幻觉检测** | 人工审查发现 | AI Reviewer + 人工 | AI Reviewer 自动 + 人工确认 | AI Reviewer 自动 + 定期审计抽检 |
| **自修复限制** | 最多 3 轮，人工确认每轮 | 最多 3 轮，自动执行 | 最多 3 轮，自动执行，超 3 轮转人工 | 最多 3 轮，自动执行，超 3 轮暂停并告警 |
| **MCP 访问** | 只读（仅读取代码和文档） | 读写（经过滤的写入操作） | 读写（过滤 + 脱敏） | 读写（过滤 + 脱敏 + 审计日志） |
| **自动合并** | 禁止 | 禁止 | 禁止 | 仅限 trivial fix |
| **Decision Point** | DP1-DP4 全部人工同步确认 | DP1-DP4 人工同步确认 | DP1-DP2 异步确认，DP3-DP4 人工同步 | DP 自动化检查 + 定期人工抽检 |

### 2.5 升级路径

从低等级升级到高等级必须满足以下所有条件，且由技术负责人或架构师批准：

```
L1 → L2 升级条件：
  ☐ 团队累计 ≥ 20 个 PR 无安全事故
  ☐ TDD 执行率 ≥ 80%
  ☐ Prompt 一次通过率 ≥ 50%
  ☐ 两层 Code Review 流程已建立
  ☐ CI Gate 已配置（编译 + 测试 + lint + SAST）
  ☐ 技术负责人批准

L2 → L3 升级条件：
  ☐ L2 稳定运行 ≥ 1 个月
  ☐ 自主成功率（PR 一次通过人工审查）≥ 70%
  ☐ 幻觉发生率 < 5%
  ☐ Self-Correction 成功率 ≥ 60%
  ☐ CI Gate 全量通过（含幻觉检测 Gate）
  ☐ 异步 Decision Point 机制已建立
  ☐ 夜间/周末 DCP 策略已定义
  ☐ 技术负责人 + 架构师联合批准

L3 → L4 升级条件：
  ☐ L3 稳定运行 ≥ 3 个月
  ☐ 自主成功率 ≥ 85%
  ☐ 每周审计通过率 ≥ 95%
  ☐ 零安全事故记录（运行期间）
  ☐ 自动化回滚机制已建立并演练
  ☐ 完整监控和告警体系已上线
  ☐ 数据分级 pre-send 扫描已启用
  ☐ 技术负责人 + 架构师 + 产品负责人三方批准
```

### 2.6 降级条件（强制）

满足以下**任一**条件，必须立即降级：

| 触发条件 | 降级目标 | 恢复条件 |
|---------|---------|---------|
| 生产安全事故（由 AI 代码导致） | 立即降到 L2 | 根因分析完成 + 修复措施验证通过 + 技术负责人批准 |
| 幻觉代码被合并到 main（未被审查发现） | 降到 L2 | 两层审查流程重建 + 幻觉检测 Gate 强化 + 技术负责人批准 |
| 审计通过率连续 2 周 < 95%（L4） | 降到 L3 | 自主成功率恢复 ≥ 85% + 连续 2 周审计通过率 ≥ 95% |
| 自主成功率连续 2 周 < 70%（L3） | 降到 L2 | 自主成功率恢复 ≥ 70% + 技术负责人批准 |
| TDD 执行率 < 80% | 降到 L1 | TDD 执行率恢复 ≥ 80% + 连续 5 个 PR 合规 |
| 密钥泄露到代码仓库 | 降到 L1 | pre-commit hook 重建 + 全面密钥扫描 + 安全团队批准 |
| AI 提供者质量严重退化（代码一次通过率 < 40%） | 降到 L2 | 切换 AI 提供者或模型 + 回归测试通过 |
| Self-Correction 连续 5 次超过 3 轮仍失败 | 降到 L2 | 根因分析 + 修复 + 连续 3 次 Self-Correction 成功 |

**降级是自动的，不需要审批。升级到高等级才需要审批。**

## 第 3 章：TDD 与质量保障

### 3.1 Red → Green → Refactor 流程

TDD 是 v5.0 的强制性质量保障机制，所有自治等级都必须遵守。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Red 阶段    │────▶│ Green 阶段   │────▶│ Refactor 阶段│
│  写测试      │     │ 写实现       │     │ 重构代码     │
│  运行 → 失败 │     │ 运行 → 通过  │     │ 运行 → 通过  │
└─────────────┘     └─────────────┘     └─────────────┘
     │                     │                     │
     │ CI 记录 Red 状态     │ CI 验证 Red→Green    │ CI 验证重构无回归
     │                     │                     │
     ▼                     ▼                     ▼
  必须失败              必须通过               必须通过
```

#### 阶段要求

| 阶段 | 要求 | CI 验证 | 人工介入 |
|------|------|--------|---------|
| **Red** | 测试文件必须先提交，且运行结果为失败 | CI 记录 Red 状态 | L1/L2：人工审核断言正确性 |
| **Green** | 实现代码提交后，所有测试通过 | CI 验证从 Red → Green 的转换 | L1：人工逐行审查；L2+：AI 自修 |
| **Refactor** | 重构后测试仍全部通过 | CI 验证无回归 | L1/L2：人工审查重构合理性 |

### 3.2 Auto-Coding 下的 TDD 流程

在 L2/L3/L4 自治等级下，AI 自动执行 TDD 流程：

```
1. AI 读取 Spec（specs/F{NNN}-{name}.md）
2. AI 从 Spec 的验收标准（Gherkin）生成测试代码
3. AI 提交测试代码 → CI 记录 Red 状态
4. [L1/L2] 人工审核测试断言是否正确
5. AI 生成实现代码
6. AI 运行测试 → 必须全部通过（Green）
7. 如有失败 → 进入 Self-Correction Loop（最多 3 轮，见 3.4 节）
8. AI 重构代码 → 测试仍通过
9. [L1/L2/L3] 人工 Code Review
10. 合并
```

#### AI 生成测试的特殊规则

| 规则 | 说明 | 检查方式 |
|------|------|---------|
| **从 Spec 生成** | 测试必须基于 Spec 中的验收标准（Gherkin） | 对照 Spec AC 条目 |
| **先于实现** | 测试必须在实现代码之前提交 | git log 提交顺序 |
| **人工审核断言** | AI 生成的断言可能错误（AI 倾向于生成"能通过"的断言而非"正确"的断言） | L1/L2：人工审核每个断言 |
| **覆盖异常路径** | AI 倾向于只生成正常路径测试 | 人工补充或 CI 检查分支覆盖率 |
| **避免过度 Mock** | AI 倾向于为一切写 Mock | 人工审查 Mock 必要性 |

### 3.3 TDD 合规检查（CI Gate）

CI 必须自动检测以下 TDD 合规项，任何一项不通过即阻塞合并：

| 检查项 | 检测方法 | 不通过后果 |
|--------|---------|-----------|
| **提交顺序** | git log 中 `_test` 文件的 commit 必须在实现代码 commit 之前 | 标记"TDD 未遵循" |
| **Red 阶段** | 测试提交后、实现提交前，测试必须失败 | 标记"TDD Red 阶段缺失" |
| **Green 转换** | 实现提交后，测试必须通过 | 阻塞合并 |
| **覆盖率阈值** | 全量运行 `go test ./... -cover` 或等价命令，每个包 ≥ 80% | 阻塞合并 |
| **连续违规** | 同一 PR 有 2 次以上 TDD 违规 | Gate 不通过 |

#### TDD 造假检测

AI 倾向于"走捷径"，可能在一次提交中同时提交测试和实现，使 TDD 流程看起来被执行但实际没有。CI 必须：

1. **检查提交粒度**：测试和实现不得在同一 commit 中
2. **验证 Red 状态**：测试提交后，在实现提交前，CI 必须记录测试失败
3. **记录转换证据**：`.gate/tdd-report.json` 中必须记录 Red → Green 的转换时间和状态

连续 2 次 TDD 未遵循 = 触发警告，并在周报中提及。

### 3.4 Self-Correction Loop（最多 3 轮，继承 v4）

AI 生成代码后，自动运行检查并自我修复。自修成功率 60-70%，超过 3 轮后急剧下降。

#### 流程

```
AI 生成代码
    │
    ▼
自动化检查（编译 + 测试 + lint）
    │
    ├─ 全部通过 → 进入 Code Review
    │
    └─ 有失败 → Self-Correction Loop
            │
            ├── 第 1 轮：AI 读取错误信息，修复代码
            │       │
            │       ├── 通过 → 进入 Code Review
            │       │
            │       └── 失败 → 第 2 轮
            │
            ├── 第 2 轮：AI 读取新的错误信息，再次修复
            │       │
            │       ├── 通过 → 进入 Code Review
            │       │
            │       └── 失败 → 第 3 轮
            │
            ├── 第 3 轮：AI 最后一次尝试修复
            │       │
            │       ├── 通过 → 进入 Code Review（但标记为"自修 3 轮后通过"）
            │       │
            │       └── 失败 → 停止，向人类报告具体问题（不进入 Code Review）
            │
            └── 超过 3 轮 → 暂停，生成详细错误报告，通知人工介入
```

#### 适用场景

| 场景 | 为什么适合 | 自修成功率 |
|------|-----------|-----------|
| Lint 错误 | AI 能理解 lint 规则并修复 | 80% |
| 类型错误 | 编译器报错明确 | 85% |
| 测试失败 | 错误信息明确，AI 能定位问题 | 70% |
| API 幻觉 | AI 生成的 API 不存在，给出错误信息后能修正 | 60% |
| 架构不合理 | 需要理解全局架构 | 20%（**不适合**自修） |

#### 核心 Prompt

Self-Correction Loop 的核心 Prompt 模板：

```markdown
以下是你自己生成的代码，运行检查发现了以下问题：

[错误信息]

请检查并修复代码，确保修复后能通过检查。
```

**为什么这个 Prompt 有效**：让 AI 检查"自己生成的代码"而非"修复代码"，触发更深入的审查。

#### 限制

- **最大轮数：3 轮**。超过 3 轮后 AI 自修质量急剧下降
- **必须记录**：每轮的错误信息和修复方案记录到 PR 描述中
- **转人工条件**：
  - 3 轮后仍未通过
  - 错误类型属于"架构不合理"等不适合自修的场景
  - 同一问题连续 3 轮自修失败

### 3.5 全量验证 Gate

AI 生成代码后，**必须**自动执行以下验证链，不得跳过：

```
1. 编译/类型检查（go build / tsc --noEmit / cargo check）
2. 全量测试（go test ./... -cover / npm test / pytest）
3. lint（golangci-lint / eslint / ruff）
4. 如果任何步骤失败 → 进入 Self-Correction Loop（最多 3 轮）
5. 3 轮后仍未通过 → 停止，向人类报告具体问题
```

#### 为什么必须全量运行

| 错误做法 | 问题 | 正确做法 |
|---------|------|---------|
| 单包测试（只测新写的包） | 遗漏跨包依赖问题 | 全量运行所有测试 |
| 抽样检查覆盖率 | 可能遗漏低分包 | 每个包必须 ≥ 80% |
| AI 声明"测试通过了" | AI 可能只运行了部分测试 | 证据文件来自 Makefile 命令执行 |

#### 覆盖率验证规则

| 规则 | 说明 |
|------|------|
| 全量运行 | 必须执行全量测试命令，不得指定单个包 |
| 每包阈值 | 每个包必须 ≥ 80%，不靠平均值掩盖低分包 |
| 证据文件 | 覆盖率数据写入 `.gate/coverage.json`，由 `make verify-gate` 生成 |
| 禁止手工填写 | 所有报告中的覆盖率数字必须引用证据文件，不得手工填写 |

#### AC 覆盖验证规则（P11 要求）

| 规则 | 说明 |
|------|------|
| AC-to-Test 映射 | 每个验收标准（AC）必须有 ≥1 对应的测试函数。映射关系写入 `.gate/ac-mapping.json` |
| 证据项要求 | 每个 AC 必须有 ≥2 项证据：代码存在证据（测试函数体存在）+ 执行证据（`.gate/test-output.json` 中该测试实际运行并通过） |
| 报告分离 | 报告时必须分别给出：包通过率（package pass rate）/ AC 覆盖率（AC coverage rate）/ 端点通过率（endpoint pass rate）。禁止合并为单一数字 |
| 完成判定 | AC 覆盖率 < 100% = 功能未完成，不得标记 Feature 为 `done` |
| 禁止推导 | 不得从"包通过率 100%"推导出"需求已满足"。包通过 ≠ AC 覆盖 |

### 3.6 AI-First-Then-Human-Refine 模式（TDD 补充）

> 作为 P3 TDD 先行的补充模式，适用于特定场景。

**默认模式是 TDD 先行（80%+ 场景）**。但在以下场景中，可以采用 AI-First-Then-Human-Refine 模式：

| 场景 | 为什么不适合 TDD | AI-First 做法 |
|------|-----------------|--------------|
| 快速原型验证 | 需求尚未稳定，测试会频繁废弃 | AI 先写可运行代码，人再理解并重构 |
| 探索性编码 | 技术方案不确定，需要先看到代码才能设计测试 | AI 生成多种方案，人选择最优后补测试 |
| UI/UX 原型 | 视觉设计需要迭代，断言不稳定 | AI 生成 UI 代码，人调整样式后再写测试 |

**规则**：
1. 使用 AI-First 模式时，必须在 PR 中说明原因
2. 代码合并前**仍然必须补齐测试**（TDD 的 Green 阶段可以延后，但不能跳过）
3. 同一模块连续使用 AI-First 模式超过 3 次，需要启动根因分析（为什么无法用 TDD？）
4. AI-First 模式下补测试的 commit 必须标记 `[TDD-DEBT]`

### 3.7 质量降级与回退机制

当 AI 输出质量持续不达标时，自动降级开发模式：

| 降级级别 | 触发条件 | 动作 | 恢复条件 |
|---------|---------|------|---------|
| **Level 0 正常** | AI 代码一次通过率 > 80% | AI 生成，人工审核 | — |
| **Level 1 警惕** | AI 代码一次通过率 60-80% | 加强审核，记录问题模式 | 连续 3 个 Feature 通过率 > 80% |
| **Level 2 降级** | AI 代码一次通过率 < 60% | 改为"人写框架，AI 填充细节" | 连续 5 个 Feature 通过率 > 70% |
| **Level 3 回退** | AI 提供者质量严重退化 | 人工编码，AI 仅辅助（注释、lint） | AI 提供者问题解决 |

**降级判定**：每周自动统计各 Feature 的 AI 代码一次通过率，自动触发对应级别。

**注意**：质量降级不同于自治等级降级（见第 2 章 2.6 节）。质量降级影响的是 AI 生成代码的信任级别，自治等级降级影响的是整个开发流程的自主程度。

## 第 4 章：AI 幻觉检测与防护（v5 重点增强）

### 4.1 什么是 AI 幻觉

AI 幻觉（AI Hallucination）是指 LLM 生成看似合理但实际错误或根本不存在的代码、API、依赖、逻辑的行为。这是概率模型的本质特征，不是 bug，而是 LLM 的固有风险。

**核心认识**：幻觉不是"偶尔发生"的问题，而是**每次生成都可能发生**的问题。Gate 的存在不是为了"检测是否发生了幻觉"，而是"确保即使发生幻觉也不会进入生产"。

### 4.2 幻觉类型清单

| 类型 | 说明 | 示例 | 危险级别 |
|------|------|------|---------|
| **API 幻觉** | 调用不存在的函数/方法/API | `user.authenticate_with_biometric()`（该 API 不存在于使用的库中） | CRITICAL |
| **依赖幻觉** | 引用不存在的依赖包或版本号 | `import { newFeature } from 'lodash@5.0.0'`（lodash 无 5.0.0） | CRITICAL |
| **逻辑幻觉** | 生成看似正确但逻辑错误的代码 | 条件判断写反、边界条件遗漏、竞态条件 | HIGH |
| **安全幻觉** | 生成看似安全但实际有漏洞的代码 | "已参数化查询"但实际仍拼接字符串 | CRITICAL |
| **测试幻觉** | 生成能通过但需求未满足的测试 | 断言写错方向（expected vs actual 颠倒） | HIGH |
| **注释幻觉** | 注释描述与实际代码行为不一致 | 注释说"处理空值"但代码未处理空值 | MEDIUM |
| **配置幻觉** | 引用不存在的配置项或环境变量 | `process.env.NON_EXISTENT_VAR` | MEDIUM |
| **协议幻觉** | 使用不存在的协议方法或端点 | REST API 调用了 gRPC 才有的方法 | HIGH |
| **覆盖率幻觉** (V04) | 把"测试文件存在且通过"等同于"需求被充分覆盖" | 声称"17/17 包通过"但 53 个 AC 中仅 15 个有执行证据 | CRITICAL |
| **完成幻觉** (C01) | 声称"完成"的程度超过了实际验证的范围 | 声称"功能基本实现"但未逐条验证 Spec 验收标准 | CRITICAL |

### 4.3 自动检测方法

#### 4.3.1 编译验证

最基础的幻觉检测——不存在的 API 和依赖会导致编译失败：

| 检测项 | 命令示例 | 检测的幻觉类型 |
|--------|---------|---------------|
| 编译/类型检查 | `go build ./...` / `tsc --noEmit` / `cargo check` | API 幻觉、类型不匹配 |
| 依赖验证 | `go mod verify` / `npm ls` / `pip freeze` | 依赖幻觉、版本幻觉 |
| 依赖安全审计 | `npm audit` / `go list -m -u` / `pip audit` | 漏洞依赖、typosquatting |

**规则**：编译失败 = 幻觉已发生且被拦截。不得通过 `--no-verify`、`--skip-checks` 等选项绕过。

#### 4.3.2 符号解析

通过 AST（抽象语法树）分析检测未定义的符号引用：

```
扫描 AI 生成的代码
    │
    ├── 解析所有函数调用
    │
    ├── 对照已定义的符号表（本地代码 + 已安装依赖）
    │
    ├── 发现未定义符号 → 标记为"潜在幻觉"
    │
    └── 全部符号已定义 → 通过
```

**实现方式**：
- TypeScript：`tsc --noEmit` 已提供完整的符号检查
- Go：`go vet` + `staticcheck` 提供额外的静态分析
- Python：`mypy --strict` 提供类型级别的符号检查
- Rust：`cargo clippy` 提供额外的 lint 和符号检查

#### 4.3.3 依赖验证

验证 AI 引入的所有依赖是否真实存在且版本正确：

| 检查项 | 检测方法 | 拦截的幻觉 |
|--------|---------|-----------|
| 依赖包存在 | `npm ls <package>` / `go list -m <package>` | 不存在的包名 |
| 版本号存在 | `npm view <package>@<version>` / `go list -m -versions <package>` | 不存在的版本号 |
| Typosquatting | 对比依赖名与已知流行包名（编辑距离检查） | 恶意包名（如 `lodahs` 替代 `lodash`） |
| 许可证合规 | `license-checker` / `go-licenses` | 不兼容的开源许可证 |

#### 4.3.4 幻觉检测 Gate 汇总表

| 幻觉类型 | 检测方法 | Gate 动作 | 自动化级别 |
|---------|---------|----------|-----------|
| 不存在的 API/函数 | 编译失败 / 符号解析失败 | **阻塞合并** | 全自动 |
| 不存在的依赖包 | `go mod verify` / `npm ls` | **阻塞合并** | 全自动 |
| 不存在的版本号 | 依赖审计 | **阻塞合并** | 全自动 |
| Typosquatting 依赖 | 编辑距离检查 + 人工审核 | **阻塞合并** | 半自动 |
| 虚构的错误处理 | 测试覆盖异常路径 | 标记为测试不足 | 全自动 |
| 注释与代码不符 | AI Reviewer 对比分析 | 标记为审查问题 | AI 辅助 |
| 逻辑错误（边界遗漏） | 分支覆盖率检查 + 人工审查 | 标记为审查问题 | AI 辅助 + 人工 |
| 安全漏洞 | SAST 扫描 + 依赖审计 | **阻塞合并**（CRITICAL 级别） | 全自动 |

### 4.4 人工检测指南

#### 4.4.1 AI Reviewer Prompt 模板

第一层审查由 AI Reviewer 自动执行：

```markdown
请审查以下 AI 生成的代码，检查以下 AI 特有问题：

## 审查清单

1. **幻觉检测**：是否存在不存在的 API/函数调用？
   - 检查所有 import/require/使用的外部库
   - 检查所有函数调用是否存在于本地代码或依赖中

2. **架构约束**：是否遵循了以下架构约束？
   [列出架构约束，来自 architecture.md 和 ADR]

3. **边界条件**：是否遗漏了以下边界条件？
   [列出边界条件，来自 Spec]

4. **安全漏洞**：是否存在以下安全问题？
   - SQL 注入（字符串拼接而非参数化查询）
   - XSS（未编码的输出）
   - 硬编码密钥/密码/token
   - 不安全的默认配置

5. **注释一致性**：注释是否与实际代码行为一致？

## 代码

[代码 diff]

## Spec

[Spec 文件相关部分]

## 审查结论

- 幻觉：[有/无]，[具体描述]
- 架构违规：[有/无]，[具体描述]
- 边界遗漏：[有/无]，[具体描述]
- 安全问题：[有/无]，[具体描述]
- 注释不一致：[有/无]，[具体描述]
```

#### 4.4.2 Human Reviewer 检查清单

第二层审查由人类执行，专注于业务逻辑和架构约束：

| 检查项 | 必须回答 | 禁止使用 |
|--------|---------|---------|
| **功能正确性** | 代码实现了 Spec 中的哪些验收标准？是否有遗漏？ | ❌ "功能基本实现" |
| **AC 覆盖验证** | 每个 AC 是否有 ≥1 对应的测试函数？是否有 ≥2 项证据（代码存在 + 实际运行输出）？ | ❌ "测试覆盖充分"（未逐条对照 AC） |
| **AI 特有风险** | 是否触发了 A01-A09 中的任何一条？具体是哪条？ | ❌ "未发现明显问题" |
| **架构一致性** | 是否符合架构文档？具体引用哪个 ADR？ | ❌ "架构上没问题" |
| **可维护性** | 代码是否清晰？有没有需要重构的地方？ | ❌ "代码质量良好" |
| **测试充分性** | 测试覆盖了哪些路径？遗漏了哪些路径？ | ❌ "测试覆盖充分" |
| **业务逻辑** | AI Reviewer 没检查到的业务逻辑是否正确？ | ❌ "业务逻辑正确"（无具体说明） |

**禁止使用模糊表述**。每个检查项必须给出具体的、可验证的结论。

**AC 覆盖验证规则**：
1. 列出 Spec 中所有 AC（如 AC-001 至 AC-005）
2. 对每个 AC，找到 ≥1 对应的测试函数名
3. 对每个 AC，确认 ≥2 项证据：代码文件存在 + `.gate/` 中有实际运行输出
4. 报告时必须分别给出：包通过率 / AC 覆盖率 / 端点通过率，禁止合并为单一数字
5. AC 覆盖率 < 100% = 功能未完成，不得标记 Spec 为 `done`

### 4.5 防护策略

#### 4.5.1 Example-Driven Prompting（继承 v4 9.2）

AI 对示例的理解远优于对规则的描述（5 倍差距）。用实际代码示例替代规则描述。

**核心做法**：在 Prompt 中引用 `.examples/` 中的示例，而非描述规则。

```markdown
# ❌ 规则描述式（效果差，容易产生幻觉）

#### 4.5.2 Prompt Chaining（继承 v4 4A）

> **前置链**：在执行 Prompt Chaining 之前，必须完成 P23 的 Requirement→Solution→Spec 链（见 1.6.2）。Prompt Chaining 用于编码执行阶段，而 Requirement→Spec 链用于编码前的方案设计阶段。

将复杂任务分解为顺序执行的 Prompt 链，每步验证输出质量。

```
分析链 → 设计链 → 编码链 → 验证链
```

| 步骤 | 输入 | 输出 | 验证方式 |
|------|------|------|---------|
| 1. 分析 | 源代码 + Spec | 需要修改的函数列表 | 人工确认分析结果 |
| 2. 设计 | 分析结果 | 设计方案 | 对照架构约束 |
| 3. 编码 | 设计方案 | 代码 | 编译 + 测试 |
| 4. 验证 | 代码 + 方案 | 验证报告 | 自动 Gate |
| 5. 修复 | 验证报告 | 修复后的代码 | 编译 + 测试 |

**链失败处理**：
- 每步验证输出，不通过则最多重试 3 次
- 超过 3 次转人工处理
- 记录失败模式到 `known-issues.yaml`

**质量提升数据**（v4 4A.4）：

| 场景 | 单 Prompt 质量 | 链式 Prompt 质量 | 提升倍数 |
|------|--------------|-----------------|---------|
| 复杂重构 | 55% | 85% | 1.5x |
| 新功能开发 | 65% | 90% | 1.4x |
| Bug 修复 | 70% | 92% | 1.3x |
| 架构变更 | 45% | 80% | 1.8x |

#### 4.5.3 Progressive Disclosure 上下文策略（继承 v4 9.1.2）

按需加载上下文，非一次性全量注入。通过优先级队列和动态替换，让 AI 始终保持高注意力。

| 优先级 | 内容 | 加载时机 |
|--------|------|---------|
| **P0（必须）** | 当前 Feature Spec、相关接口契约 | 每次对话开始 |
| **P1（应该）** | 相关源文件、架构约束 | 首次提到该文件时 |
| **P2（可选）** | 商业计划、竞品分析、项目历史 | 人手动注入时 |

**动态替换规则**：
1. 对话中不再相关的 P1 上下文应被移除，为新上下文腾出空间
2. 当 AI 进入新的子任务时，自动加载该子任务相关的 P1 上下文
3. P0 上下文在整个对话中保持不变

## 第 5 章：Spec 驱动开发

### 5.1 为什么需要 Spec 驱动

AI 不是人——它不会"理解意图"，只会"按字面执行"。没有明确的 Spec，AI 的代码生成就是盲人摸象：可能正确，可能完全错误，取决于概率。

**Spec 是 AI 开发的大脑导航图**。没有 Spec 的 AI Coding 就像没有需求的传统开发——产出完全不可预测。

### 5.2 Feature Spec 规范

每个 Feature 必须有且仅有一个 Spec 文件，存放在 `specs/` 目录：

```
specs/
├── F001-user-registration.md
├── F002-user-login.md
├── F003-password-reset.md
└── ...
```

#### 5.2.1 Spec 文件结构

```yaml
---
type: feature-spec
id: F001
name: "用户注册"
version: 1.0
status: draft | validated | ready | in-progress | done | deprecated
priority: must | should | could | wont
created: 2026-04-14
updated: 2026-04-14
validated_by: "@name"
validated_at: 2026-04-14
references:
  - PRD: ipd/03-detailed-planning/PRD.md
  - architecture: ipd/03-detailed-planning/architecture.md
required_context:
  - ipd/03-detailed-planning/architecture.md#L50-L80
  - src/auth/models.py
ai_cost_budget: 5.00  # USD
ai_model_tier: L3
accessibility: WCAG 2.1 AA
i18n: required
autonomy_level: L2  # 本 Feature 使用的自治等级
---
```

```markdown
# Feature: F001 用户注册

## 上下文
- 所属模块：认证模块
- 架构约束：使用 OAuth2 协议，JWT 认证
- 非功能需求：
  - 性能：注册接口 P99 < 500ms
  - 安全：密码加密（bcrypt），防止枚举攻击
  - 无障碍：WCAG 2.1 AA
  - 国际化：支持中/英/日文

## 用户故事
### US-001: 新用户注册
**Given** 用户在注册页面
**When** 用户输入邮箱、密码并点击注册
**Then** 系统发送验证邮件，用户状态为"待验证"

## 验收标准
### AC-001: 正常注册流程
```gherkin
Given 用户未注册
When 用户输入有效邮箱、密码(>=8字符，包含大小写+数字)
Then 返回 201 Created
And 发送验证邮件
And 用户状态为 pending_verification
```

### AC-002: 重复注册
```gherkin
Given 邮箱已注册
When 用户尝试用相同邮箱注册
Then 返回 409 Conflict
And 消息提示"该邮箱已注册"
```

### AC-003: 密码强度不足
```gherkin
Given 用户在注册页面
When 用户输入弱密码（如 "password123"）
Then 返回 400 Bad Request
And 消息提示"密码强度不足，需要至少 8 字符且包含大小写字母和数字"
```

## 边界条件
- 输入为空时：400 Bad Request，具体字段错误提示
- 输入非法时：400 Bad Request，不允许 SQL 特殊字符
- 并发场景：同一邮箱同时注册 → 串行处理，返回适当消息
- 邮箱格式：RFC 5322 标准校验
- 速率限制：同一 IP 每小时最多 10 次注册尝试

## 数据模型
```
User {
  id: UUID
  email: string (unique, indexed)
  password_hash: string (bcrypt)
  status: enum[pending_verification, active, suspended]
  created_at: timestamp
  updated_at: timestamp
}
```

## API 设计
```
POST /api/v1/auth/register
Request: { email, password }
Response 201: { id, email, status }
Response 400: { error, field }
Response 409: { error }
Response 429: { error, retry_after }
```

- [ ] Spec 已通过人工验证（validated_by 已填写）
- [ ] 测试已编写并通过
- [ ] 实现代码已编写
- [ ] Code Review 通过（两层审查）
- [ ] 安全扫描通过
- [ ] a11y 检查通过（如适用）
- [ ] i18n 检查通过（如适用）
```

### 5.3 Spec Validation Gate（Spec 验证门禁）

Spec 编写完成后、驱动开发之前，必须通过验证。Spec 状态从 `draft` 变更为 `validated` 后才能进入开发。

| 验证项 | 验证方式 | 执行人 | 检查要点 |
|--------|---------|-------|---------|
| **与 PRD 一致性** | Spec 的验收标准是否覆盖 PRD 中的需求 | 人工审核 | 每个 PRD 需求至少有一个 AC 对应 |
| **与架构一致性** | Spec 是否遵循架构约束（技术选型、模块划分、安全设计） | 人工审核（架构师或技术负责人） | 不违反任何 ADR |
| **验收标准可测试** | 每个 AC 能否转化为自动化测试 | 技术负责人 | Gherkin 格式可执行 |
| **边界条件完整** | 是否覆盖空值、非法值、并发、速率限制等场景 | 人工审核 | 至少覆盖正常路径 + 3 种异常路径 |
| **非功能需求明确** | 性能、安全、a11y、i18n 是否声明 | 人工审核 | 数值化，不可用"足够快"等模糊描述 |
| **事实性声明准确** | Spec 中的技术描述是否准确（如"使用 OAuth2"） | 领域专家 | 引用正确的协议和库版本 |

### 5.4 Auto-Coding 下的 Spec

在 L3/L4 自治等级下，Spec 是 AI Agent 自主获取任务的来源：

```
specs/ 目录
    │
    ├── status: ready 的 Spec
    │       │
    │       ▼
    │   AI Agent 读取 Spec
    │       │
    │       ▼
    │   自动执行开发循环
    │       │
    │       ▼
    │   Spec 状态 → in-progress → done
    │
    └── status: validated 的 Spec（等待被获取）
```

#### Auto-Coding Spec 规则

| 规则 | 说明 |
|------|------|
| **Spec 必须预先验证** | 即使是 L4 自主编码，Spec 在放入 `specs/` 队列前必须经过人工验证（`validated` 状态） |
| **AI 不得自行创建 Spec** | AI 可以辅助生成 Spec 草稿，但 Spec 的 `validated` 状态必须由人工设置 |
| **Spec 优先级** | AI 按 `priority` 字段排序处理：must > should > could > wont |
| **依赖检测** | AI 在开始开发前检查 Spec 间的依赖关系，按依赖 DAG 排序执行 |
| **状态自动更新** | AI 开始开发时自动将 Spec 状态设为 `in-progress`，完成后设为 `done` |
| **冲突检测** | 如果多个 Spec 修改同一文件，AI 必须报告冲突并暂停，等待人工裁决 |

### 5.5 Spec Validation Gate 自动化（CI Gate）

CI 自动检查以下 Spec 合规项：

| 检查项 | 检测方法 | 不通过后果 |
|--------|---------|-----------|
| Spec 文件存在 | PR 描述中必须引用 Spec 文件 | 阻塞合并 |
| Spec 状态为 validated/ready | 检查 YAML frontmatter 中的 status 字段 | 阻塞合并 |
| 验收标准有 Gherkin 格式 | 解析 Spec 文件中的 gherkin 代码块 | 标记为 Spec 不合格 |
| 非功能需求已声明 | 检查 Spec 中的非功能需求章节 | 标记为 Spec 不合格 |
| validated_by 已填写 | 检查 YAML frontmatter | 阻塞合并 |
| **问题定义已填写** | 解析 `problem_statement` 字段，非空 | 阻塞合并 |
| **来源追溯有效** | `origin` 字段非空且为合法值（PRD / user-feedback / poc-assumption / tech-debt） | 阻塞合并 |
| **PRD 映射检查** | 当 `origin` 包含 PRD 时，`prd_user_story_refs` 不得为空 | 阻塞合并 |

#### 实验性 Feature 管控

POC 验证假设、技术探索类需求产生的 Spec 与正式功能不同：它们的目标是"验证"不是"交付"。

| 规则 | 说明 |
|------|------|
| 标记 | `origin: poc-assumption` 或 `origin: tech-exploration` 的 Spec 必须设 `experimental: true` |
| 分支限制 | 实验性 Spec 产生的代码只能合入 `experimental/*` 分支，不得合入 `main` |
| 提升为正式 | 实验验证通过后，人工更新 Spec：移除 `experimental`、补充 `prd_user_story_refs`、填写 `problem_statement`，方可合入 main |
| 时效性 | 实验性 Spec 创建 30 天后未提升为正式则自动归档 |

#### 自动生成 Skill 的质量要求

当 feature 涉及 LLM 自动生成可复用的 skill/tool/能力时，Spec 的验收标准必须覆盖以下维度，不得仅验证"结构合法"：

| 维度 | 要求 | 验证方式 |
|------|------|---------|
| **名称泛化** | skill.name 不得绑定具体实例（地名、版本号、用户ID等），必须为通用语义标识 | AC 中明确禁止实例化命名 |
| **别名覆盖** | skill.aliases 不得为空，必须包含原始意图的多种表达方式和同义变体 | AC 中要求 aliases ≥ 3 且包含泛化表达 |
| **相似意图匹配** | 生成的 skill 必须能被语义相似的未知意图匹配到，而非仅精确匹配原始意图 | AC 中提供 ≥2 个变体意图并验证匹配 |
| **标签泛化** | skill.tags 必须包含通用领域关键词，不得包含具体实例标识 | AC 中检查 tags 不含地名/人名/版本号 |
| **质量评分** | 不得硬编码静态分数，必须基于实际能力动态计算 | AC 中要求 quality_score 有计算依据 |

## 第 6 章：Prompt 管理

### 6.1 Prompt 版本化（P9 详解）

Prompt 是 AI 时代的需求文档，必须版本化、可追溯、可复现。

#### 6.1.1 Prompt 文件结构

每个用于生成代码、测试或重要交付件的 Prompt 必须存储在 `prompts/` 目录：

```
prompts/
├── P001-user-reg-test.md
├── P002-user-reg-impl.md
├── P003-user-login-test.md
└── ...
```

Prompt 文件的 YAML frontmatter：

```yaml
---
id: P001
name: "用户注册 - 测试代码生成"
version: 1.0
created: 2026-04-14
model: qwen3.5-plus
temperature: 0.0
top_p: 1.0
purpose: test-generation
feature: F001
input_context:
  - specs/F001-user-registration.md
  - ipd/03-detailed-planning/architecture.md
output_artifacts:
  - tests/test_user_registration.py
status: active  # active | deprecated | retired
quality_metrics:
  first_pass_rate: 85%
  avg_retry_count: 0.5
  output_consistency: high
---
```

#### 6.1.2 Prompt-输出追溯

每个 PR 必须在描述中声明使用的 Prompt 版本：

```
PR #123: 用户注册功能

Feature: F001
Prompt: P001 v1.0 (prompts/P001-user-reg-test.md)
Model: qwen3.5-plus, temperature=0.0
Spec: specs/F001-user-registration.md v2.1
Autonomy Level: L2
```

**追溯的价值**：
- **复现**：同一 Prompt + 同一模型 + 同一温度 = 可复现的输出
- **调试**：当 AI 输出质量下降时，可以追溯到具体 Prompt 版本
- **审计**：合规要求下，可以证明代码是由什么 Prompt 生成的
- **回滚**：当 Prompt 质量退化时，可以回退到上一版本

#### 6.1.3 Prompt 质量评估

每次 Prompt 执行后记录：

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| 一次通过率 | 输出通过 Review 的比例 | 评估 Prompt 质量 |
| 重试次数 | 需要重新生成的次数 | 评估 Prompt 精确度 |
| 输出一致性 | 同 Prompt 多次运行的 diff | 评估模型确定性 |

**Prompt 一次通过率 < 50% = Prompt 需要重写。**

### 6.2 Prompt Chaining（继承 v4 4A）

> **前置链**：在执行 Prompt Chaining 之前，必须完成 P23 的 Requirement→Solution→Spec 链（见 1.6.2）。Prompt Chaining 用于编码执行阶段，而 Requirement→Spec 链用于编码前的方案设计阶段。

将复杂任务分解为顺序执行的 Prompt 链，每步的输入是上一步的输出。质量比单个 Prompt 提升 3-5 倍。

#### 典型链

```
分析链 → 设计链 → 编码链 → 验证链
```

| 步骤 | Prompt 示例 | 输入 | 输出 |
|------|-----------|------|------|
| 1. 分析 | "分析 [文件] 的结构，列出需要修改的函数" | 源代码 | 函数列表 |
| 2. 设计 | "基于上述分析，设计修改方案" | 分析结果 | 设计方案 |
| 3. 编码 | "按照方案编写代码" | 设计方案 | 代码 |
| 4. 验证 | "检查代码是否完整实现了方案" | 代码 + 方案 | 验证报告 |
| 5. 修复 | "修复验证中发现的问题" | 验证报告 | 修复后的代码 |

#### 链配置（`.aicoding.yaml`）

```yaml
prompt_chaining:
  max_retries_per_step: 3
  steps:
    - analyze    # 分析代码结构
    - design     # 设计方案
    - code       # 编写代码
    - verify     # 验证
    - fix        # 修复
  fail_action: escalate_to_human  # 链失败时转人工
```

### 6.3 Example-Driven Prompting（继承 v4 9.2）

AI 对示例的理解远优于对规则的描述（5 倍差距）。

#### 使用方式

```markdown
# ❌ 规则描述式（效果差）
请确保代码遵循单一职责原则，一个函数只做一件事。

# ✅ 示例驱动式（效果好）
请参考以下示例的函数拆分方式：

示例：.examples/architecture/repository-pattern.md

请按照相同的风格编写。
```

#### 示例维护

- 每个示例必须包含：**问题描述**、**错误示例**、**正确示例**、**为什么这样做好**
- 示例必须来自**真实项目代码**，而非 AI 生成
- 每季度审查示例库，更新过时的示例

### 6.4 Progressive Disclosure 上下文策略（继承 v4 9.1.2）

按需加载上下文，通过优先级队列和动态替换，让 AI 始终保持高注意力。

| 优先级 | 内容 | 加载时机 |
|--------|------|---------|
| **P0（必须）** | 当前 Feature Spec、相关接口契约 | 每次对话开始 |
| **P1（应该）** | 相关源文件、架构约束 | 首次提到该文件时 |
| **P2（可选）** | 商业计划、竞品分析、项目历史 | 人手动注入时 |

#### 上下文 Manifest

每个 Feature Spec 必须声明 `required_context`，精确列出 AI 需要的文件和范围：

```yaml
required_context:
  - file: ipd/03-detailed-planning/architecture.md
    lines: 50-80  # 仅加载接口设计部分
  - file: src/auth/models.py
    lines: 1-120  # 完整加载用户模型
  - file: specs/F001-user-registration.md
    lines: all    # 完整加载当前 Spec
```

**规则**：
- 上下文总量必须在模型窗口限制内（预留 50% 给对话内容）
- 禁止全量加载 architecture.md 或 PRD.md，只加载相关段落
- 超出上下文窗口时，必须拆分任务

### 6.5 Auto-Coding 下的 Prompt 持久化

在 L3/L4 自治等级下，Prompt 必须自动持久化：

| 要求 | 说明 |
|------|------|
| **自动存储** | 每次 AI 生成代码时，使用的 Prompt 自动保存到 `prompts/` 目录 |
| **自动关联** | Prompt 自动关联到 Feature Spec 和 PR |
| **自动版本化** | 同一 Prompt 的每次修改都递增版本号 |
| **定期审查** | 每季度审查 Prompt 仓库，删除过时或不再使用的 Prompt |
| **回归测试** | 模型更新或 Prompt 修改后，用标准 Prompt 套件测试输出质量，质量下降 > 15% 则回退 |

#### Prompt 安全

| 规则 | 说明 |
|------|------|
| **禁止动态加载外部 Prompt** | Prompt 文件必须来自本地仓库，禁止从 URL 或外部服务加载 |
| **Prompt 注入防护** | Spec 中的用户输入不得直接拼接到 Prompt 中，必须经过转义和过滤 |
| **Prompt 审计** | 每季度审查 Prompt 仓库，删除过时或不再使用的 Prompt |

## 第 7 章：人与 AI 的分工

### 7.1 基础分工

| 活动 | 人类角色 | AI 角色 | 自治等级差异 |
|------|---------|---------|-------------|
| 战略决策（投什么） | **决策者** | 信息整理、数据分析 | 所有等级：人类决策 |
| 产品定义（做什么） | **决策者** | 需求结构化、场景建模 | 所有等级：人类决策 |
| 架构设计（怎么做） | **决策者** | 方案对比、影响分析 | 所有等级：人类决策 |
| Spec 编写 | **决策者 + 审核者** | 草稿生成、格式结构化 | L1：人工主导；L2+：AI 草稿，人工审核 |
| 测试编写 | **审核者** | **执行者**（基于 Spec 生成） | L1：人工审核每个断言；L2+：AI 生成，人工抽检 |
| 编码实现 | **审核者** | **执行者**（Spec 驱动） | L1：人工逐步引导；L2+：AI 自主执行 |
| 文档生成 | **审核者** | **执行者** | L1/L2：人工审核；L3/L4：AI 自主 |
| 安全审查 | **决策者** | 辅助扫描、模式识别 | 所有等级：人类最终决策 |
| 部署运维 | **监督者** | **执行者**（自动化） | L1/L2：人工批准部署；L3：AI 可部署 dev/staging；L4：同 L3 |

**核心原则**：AI 放大人的执行力，不替代人的判断力。

### 7.2 各自治等级下的分工差异

#### L1：辅助编码

```
人 ←── 每一步确认 ──→ AI
  │                      │
  ├── 编写 Spec 草稿      │
  ├── 审核 Spec           │
  ├── 编写测试断言        │
  ├── 审核测试            │
  ├── 引导代码生成        │
  ├── 逐行审查代码        │
  ├── 确认自修结果        │
  ├── 执行 Code Review    │
  └── 执行合并            │
```

#### L2：半自主编码

```
人 ←── PR 审查 ──→ AI（自主开发循环）
  │                      │
  ├── 验证 Spec           ├── 读取 Spec
  ├── [等待 AI 执行]      ├── 生成测试
  ├── 审核断言            ├── 提交测试（Red）
  ├── 两层 Code Review    ├── 生成实现
  └── 执行合并            ├── 运行测试（Green）
                          ├── Self-Correction（最多 3 轮）
                          ├── 全量验证 Gate
                          └── 创建 PR
```

#### L3：受限自主编码

```
人（早班审核者）            AI（夜班开发者）
  │                          │
  ├── 夜间前 DCP 确认        ├── 从 specs/ 读取 ready 任务
  ├── 异步确认 DP1/DP2       ├── 自主执行开发循环
  ├── 批量审查 PR            ├── 自动创建 PR
  └── 合并 PR                └── 更新 Spec 状态
```

#### L4：完全自主编码

```
人（周审计者）                AI（独立开发者）
  │                          │
  ├── 定义商业目标           ├── 从 specs/ 读取任务
  ├── 验证 Spec（定期抽检）  ├── 自主执行全部开发循环
  ├── 周审计（≥10% 抽样）    ├── 自动合并 trivial fix
  └── 处理降级事件           └── 自动创建 PR 非 trivial 变更
```

### 7.3 Decision Point 介入模式（继承 v4 3D）

定义 7 个关键决策点，人在这些点必须介入确认，拦截 90%+ 的方向性错误。

#### 决策点清单

| # | 决策点 | 时机 | 人需要确认什么 | 拦截的错误类型 |
|---|--------|------|---------------|---------------|
| **DP0** | 需求确认点 | 需求分析完成、进入架构适配前 | "这个需求理解正确吗？优先级对吗？" | 需求理解偏差、优先级错误 |
| **DP0.5** | 架构方向点 | 架构适配分析完成、进入方案设计前 | "架构方向与现有系统一致吗？有无冲突？" | 架构与现有系统不兼容、技术选型错误 |
| **DP0.7** | 方案可行性点 | 方案设计完成、进入 Spec 生成前 | "这个方案可行吗？有没有遗漏的风险？" | 方案不可行、遗漏关键约束 |
| **DP1** | 需求理解点 | AI 分析完代码结构和需求后 | "你理解的对吗？这个方案是我要的吗？" | AI 误解需求、遗漏关键约束 |
| **DP2** | 架构方案点 | AI 给出技术方案后 | "这个架构方向对吗？有没有更好的方案？" | AI 选择不合适的模式、过度工程 |
| **DP3** | 发布决策点 | 所有 Feature 完成、准备发布前 | "产品真的准备好了吗？" | 功能不完整、体验不佳、遗漏边缘场景 |
| **DP4** | 紧急变更点 | Hotfix 或紧急需求时 | "这个修复方向对吗？有没有副作用？" | 修复引入了新问题、方向错误 |

#### Decision Point 执行流程

```
AI 完成当前阶段 → AI 输出摘要（做了什么、为什么、有什么风险） → 人确认 → AI 继续 / 人修正方向
```

**每个 Decision Point 的 Prompt 模板**：

```markdown
## Decision Point: [DP名称]

AI 已完成 [阶段描述]。

### AI 摘要
- 做了什么：
- 为什么这样做：
- 有哪些风险和备选方案：

### 请确认
1. 方向是否正确？
2. 有无遗漏的关键约束？
3. 是否建议调整？

确认后可继续执行下一步。
```

### 7.4 异步 Decision Points（用于夜间/周末开发）

在 L3 受限自主编码模式下，AI 在夜间或周末自主开发时，人类审核者可能不在场。此时使用异步 Decision Point：

```
AI 到达 Decision Point
    │
    ├── 生成 DP 摘要（做了什么、为什么、风险）
    │
    ├── 通过通知渠道（Slack/Email/IM）发送给人工审核者
    │
    ├── 暂停开发（等待确认）  或  继续开发（非阻塞模式）
    │
    └── 人工审核者确认 → AI 继续 / 人修正方向
```

#### 异步 DP 模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **阻塞模式** | AI 到达 DP 后暂停，等待人工确认 | DP1（需求理解）、DP2（架构方案） |
| **非阻塞模式** | AI 通知人工，继续执行，人工确认后如不认可则 revert | DP3（发布决策），用于低风险变更 |
| **延迟模式** | AI 记录 DP 摘要，第二天人工审查时一并确认 | 多个低风险 DP 的批量确认 |

#### 异步 DP 规则

| 规则 | 说明 |
|------|------|
| **DP1/DP2 必须阻塞** | 需求理解和架构方案是方向性决策，必须在执行前确认 |
| **超时自动降级** | 如果人工在 2 小时内未确认，AI 自动降级为"非阻塞模式"并记录警告 |
| **通知必须有完整上下文** | 通知中包含 Spec 引用、变更文件列表、风险评估 |
| **确认必须具体** | 人工不得使用"OK"、"继续"等模糊回复，必须回答 3 个确认问题 |

### 7.5 Decision Point 与 DCP 的关系

| | DCP（决策门） | Decision Point（决策点） |
|---|---|---|
| 粒度 | 阶段级（整个 Phase） | 任务级（单个 Feature/函数） |
| 频率 | 每个 Phase 一次 | 每个开发任务多次 |
| 参与者 | 产品负责人 + 架构师 | 开发者本人 |
| 目的 | 商业和技术方向判断 | 执行方向判断 |
| L3/L4 执行 | 异步或自动化 | 异步通知 + 人工确认 |
| 关系 | 宏观保护 | 微观保护 |

**两者互补，不可替代。**

### B.2 自治等级速查

```
L1 辅助编码 ── 人每一步参与，新手团队

### B.3 TDD 速查

```
Red → Green → Refactor
 │       │        │
 │       │        └─ 测试仍通过 = 重构正确
 │       └─ 测试全通过 = 实现正确
 └─ 测试必须失败 = 测试有意义

AI TDD：AI 生成测试 → 验证 Red → AI 生成实现 → 验证 Green → AI 自修（≤3 轮）
CI Gate：提交顺序 + Red 记录 + Green 验证 + 覆盖率 ≥ 80%/包
```

### B.4 幻觉检测速查

```
幻觉类型：API 幻觉 | 依赖幻觉 | 逻辑幻觉 | 安全幻觉 | 测试幻觉 | 注释幻觉 | 覆盖率幻觉(V04) | 完成幻觉(C01)
检测方法：编译验证 → 符号解析 → 依赖验证 → AI Reviewer → Human Reviewer → AC 映射验证
防护策略：Example-Driven Prompting + Prompt Chaining + Progressive Disclosure
应急流程：检测 → 阻止合并 → 分析类型 → 修复 → 强化 Gate → 记录
```

## 附录 E：AI 代码审查清单（A01-A09 详解）

> 本节详细展开 v4 7.3 节中的 AI 代码特殊规范 A01-A09，每条都附具体的审查方法和示例。

### A01：禁止接受不存在的 API 调用

**问题**：AI 会幻觉出不存在的库函数、方法或 API 端点。

**审查方法**：
1. 检查所有 import/require/use 语句，确认包/模块确实存在
2. 检查所有函数调用，确认存在于本地代码或已安装依赖中
3. 使用 IDE 的"跳转到定义"功能快速验证

**示例**：
```python
# ❌ 幻觉：requests 库没有 retry_session 方法
session = requests.retry_session(retries=3)

# ✅ 正确：使用 requests.Session 配合 urllib3.Retry
from requests.adapters import HTTPAdapter
from urllib3.util.retry import Retry
session = requests.Session()
retry = Retry(total=3, backoff_factor=0.5)
adapter = HTTPAdapter(max_retries=retry)
session.mount('https://', adapter)
```

### A02：禁止接受虚构错误处理

**问题**：AI 倾向于写空 catch 块或捕获根本不存在的异常类型。

**审查方法**：
1. 每个 try/catch 必须有具体的异常类型（不得 bare except）
2. 每个 catch 块必须有实际处理（日志、重试、降级），不得空 catch
3. 验证 catch 的异常类型确实可能被抛出

**示例**：
```python
# ❌ 虚构错误处理：空 catch + 捕获不存在的异常
try:
    result = api.call()
except APINotImplementedError:  # 该异常不存在
    pass  # 空 catch

# ✅ 正确：具体异常 + 实际处理
try:
    result = api.call()
except requests.exceptions.ConnectionError as e:
    logger.warning(f"API 连接失败: {e}")
    raise ServiceUnavailableError from e
```

### A03：审查依赖版本

**问题**：AI 可能引用不存在的版本号或已废弃的包。

**审查方法**：
1. 运行 `npm ls` / `go mod verify` / `pip check` 验证依赖完整性
2. 检查 `package.json` / `go.mod` / `requirements.txt` 中的版本号是否存在
3. 使用 `npm audit` / `trivy` / `pip audit` 检查安全漏洞

### A04：审查注释准确性

**问题**：AI 可能生成与实际代码行为不符的注释。

**审查方法**：
1. 每个函数注释必须描述函数实际做什么，而非"应该做什么"
2. 检查注释中的参数说明、返回值说明是否与代码一致
3. 特别注意"安全相关"注释（如"已防止 SQL 注入"）是否属实

### A05：检查过度工程

**问题**：AI 倾向于生成比需求复杂 10 倍的方案。

**审查方法**：
1. 每个类/函数问："这个抽象是必要的吗？能用更简单的方式实现吗？"
2. 检查是否存在"为未来扩展预留"但当前不需要的代码
3. 圈复杂度 > 10 的文件需要特别审查

**反模式信号**：
- 为 3 行逻辑创建了 5 个类
- 使用了 3 层以上的设计模式
- 接口只有一个实现
- 工厂模式只生产一种产品

### A06：检查边界条件

**问题**：AI 倾向于只写正常路径，遗漏边界条件。

**审查方法**：
1. 对照 Spec 中的边界条件清单，逐一验证是否实现
2. 检查函数入口是否有参数校验（null、空字符串、非法值）
3. 检查集合操作是否有空集合处理
4. 检查数值计算是否有溢出/下溢处理

### A07：检查隐藏假设

**问题**：AI 代码可能隐含未声明的假设。

**审查方法**：
1. 识别代码中"如果 X 不成立就会崩溃"的地方
2. 确认每个假设要么在 Spec 中声明，要么在代码中防御
3. 特别注意时间假设（"这个 API 永远返回 200"）、数据假设（"这个字段永远存在"）

### A08：确认架构约束理解

**问题**：AI 可能忽略架构文档中的约束。

**审查方法**：
1. 对照 `architecture.md` 和 `adr/` 中的决策，验证代码是否遵循
2. 检查是否引入了架构中禁止使用的库或模式
3. 检查模块间依赖方向是否正确（不得循环依赖）

### A09：检查 AI 特有模式不一致

**问题**：同一功能多次生成可能使用不同模式。

**审查方法**：
1. 检查同一模块中相似功能的实现方式是否一致
2. 检查错误处理模式是否统一（不得混用 try/catch、if err != nil、Optional）
3. 检查命名风格是否一致

## 附录 I：幻觉检测详细场景

### I.1 场景 1：AI 使用了不存在的第三方库方法

**触发**：AI 生成的代码调用了 `requests_retry_session()`，但 requests 库无此方法。

**检测链**：
```
编译失败（tsc --noEmit / mypy / go build）
    → CI 记录 "类型错误：模块无此属性"
    → 触发 Self-Correction Loop 第 1 轮
    → AI 尝试修复，可能再次幻觉
    → 第 2 轮、第 3 轮
    → 3 轮后仍失败 → 转人工
```

**修复**：人工查阅 requests 文档，找到正确的重试实现方式，更新代码。

### I.2 场景 2：AI 引用了虚构的配置参数

**触发**：AI 使用 `server.max_concurrent_connections`，但实际配置项是 `server.max_connections`。

**检测链**：
```
运行时错误（仅在测试运行时暴露）
    → 测试失败：KeyError: 'max_concurrent_connections'
    → 触发 Self-Correction Loop
    → AI 可能修正为正确的配置名
    → 通过或转人工
```

**防护**：在 Prompt 中引用配置文件 schema，或使用 Example-Driven Prompting 提供正确示例。

### I.3 场景 3：AI 生成了能通过但需求未满足的测试

**触发**：Spec 要求"密码必须包含大小写字母和数字"，AI 生成的测试断言是 `assert response.status == 201`（只检查了返回码，未检查密码策略）。

**检测**：
```
Human Reviewer 检查：
    → 对照 Spec 的 AC 条目
    → 发现测试未覆盖 AC-003（密码强度不足）
    → 标记"测试不充分"
    → 要求补充测试
```

**防护**：Human Reviewer 必须逐条对照 Spec 的验收标准检查测试覆盖。

### I.4 场景 4：AI 的注释声称做了安全检查但实际没有

**触发**：代码注释写"已防止 SQL 注入"，但实际代码使用字符串拼接。

```python
# ❌ 注释声称安全，实际不安全
def get_user(email):
    # 已防止 SQL 注入：使用参数化查询
    query = f"SELECT * FROM users WHERE email = '{email}'"  # SQL 注入！
    return db.execute(query)
```

**检测**：
```
AI Reviewer：
    → 对比注释和代码行为
    → 发现不一致
    → 标记"注释与代码不符"

Human Reviewer：
    → 审查代码逻辑
    → 发现 SQL 拼接
    → 触发 A04（注释准确性）+ S04（SQL 拼接）
```

### I.5 场景 5：AI 引入了 typosquatting 依赖

**触发**：AI 引入 `lodahs` 而非 `lodash`。

**检测链**：
```
依赖验证 Gate：
    → npm install lodahs 成功（因为 lodahs 确实存在）
    → 编辑距离检查：lodahs 与 lodash 编辑距离 = 1
    → 标记"潜在 typosquatting"
    → 阻塞合并，等待人工确认
```

**防护**：CI 中启用依赖名编辑距离检查，与流行包名对比。
